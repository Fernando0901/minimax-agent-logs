## 3. Causa Raíz

### Versión accesible (para Fernando)

La función `generate_image()` fue implementada correctamente el 2026-04-04 por el MiniMax-Agent como parte de su auto-mejora. Sin embargo, cuando se agregó, solo se conectó al comando `/imagine` de Telegram (`main.py`). **Se olvidó registrar la herramienta en los dos lugares de `brain.py` que son necesarios para que MiniMax M2.7 la conozca y pueda invocarla:**

1. `build_tools_list()` — declara el "schema" de la herramienta para que el LLM sepa que existe
2. `execute_tool_call()` — routing que conecta el nombre de la herramienta con la función real

Sin ambas inscripciones, la herramienta existe en el código pero el modelo de lenguaje no tiene forma de saberlo. El bot arranca normalmente, pero cuando Fernando pide una imagen en lenguaje natural, MiniMax M2.7 le explica que no puede generar imágenes en lugar de simplemente invocarla.

### Versión técnica

```
image_handler.py: generate_image() — ✓ Implementada (MiniMax Image-01 API, rate limiting, usage tracking)
brain.py: build_tools_list() — ✗ Faltaba el schema de generate_image
brain.py: execute_tool_call() — ✗ Faltaba el handler/routing para generate_image
```

**Mecanismo de error:** El mismo patrón que ya había ocurrido con `get_overdue_activities` en sesiones anteriores (fix_history session 20260406_015053) — una capability existe en un módulo helper pero no se registra en los dos puntos de entrada del tool-calling en `brain.py`. No había validación arquitectónica que obligara a registrar una herramienta en ambos lugares simultáneamente.

## 4. Qué se Corrigió

### Paso 1 — Schema en build_tools_list()
Se agregó el tool definition de `generate_image` a la lista `minimax_tools` en `build_tools_list()` de `brain.py`, con el schema OpenAPI correspondiente (parámetro `prompt` tipo string, description completa, rate limit documentado).

### Paso 2 — Routing en execute_tool_call()
Se agregó el bloque de routing `if tool_name == "generate_image":` en `execute_tool_call()` que:
- Llama a `image_handler.generate_image()` con el prompt
- Registra en logs (INFO/WARNING/ERROR según el resultado)
- Serializa el resultado como JSON con `url` y `local_path`
- Maneja el caso de rate limit ("Límite" en el mensaje de error)

### Paso 3 — _TOOL_REGISTRY + _validate_tool_registry()
Se agregó un registry formal y una validación arquitectónica:
- `_TOOL_REGISTRY: dict` a nivel módulo que mapea tool name → handler function
- `_validate_tool_registry()` que corre al momento de importar `brain.py`
- Verifica que cada schema en `build_tools_list()` tenga un handler correspondiente
- Si hay desbalance, el container hace **fail-fast con crash loop** y log CRITICAL

Esta validación previene recurrencia: si en el futuro alguien agrega una herramienta a `build_tools_list()` pero olvida el handler, el bot no arranca — no hay falla silenciosa.

## 5. Mejoras Estructurales Añadidas

### Validación startup (fail-fast)
`_validate_tool_registry()` corre al importar `brain.py`. Si alguien agrega una herramienta a `build_tools_list()` pero olvida registrarla en `execute_tool_call()`, el container entra en crash loop con mensaje `CRITICAL: TOOL_REGISTRY_GAP`. No hay falla silenciosa de runtime.

### Dual-registration pattern documentado
El patrón de registrar herramientas en DOS lugares (`build_tools_list()` + `execute_tool_call()`) + `_TOOL_REGISTRY` como single source of truth + `_validate_tool_registry()` como guardián startup está ahora documentado en:
- `agent_memory.md` (memoria persistente del agente)
- `pipeline/CLAUDE.md` Known Failure Patterns (ocurrence count incremented a 2)
- CHANGELOG.md (entrada 2026-04-07)

### Tests de integración
4 tests automatizados en `tests/test_image_gen_tool_integration.py`:
1. `test_generate_image_schema_in_build_tools_list` — schema existe con parámetro prompt
2. `test_generate_image_routing_executes` — routing devuelve url o error de rate limit
3. `test_validate_tool_registry_passes` — validación startup pasa sin AssertionError
4. `test_tool_registry_integrity` — _TOOL_REGISTRY es dict, generate_image registrada, callable
