## 3. Causa Raíz

### Versión accesible para Fernando
El bot usaba `reply_photo()` con la URL remota del servidor OSS de MiniMax. Cuando Fernando pedía una imagen con `/imagine`, el servidor de MiniMax guardaba la imagen en su propio almacenamiento en la nube (OSS) y devolvía un link. El bot enviaba ese link a Telegram, que intentaba descargar la imagen desde el servidor OSS de MiniMax. Esto fallaba silenciosamente porque el link era inaccesible o expiraba.

### Detalle técnico
La función `generate_image()` en `image_handler.py` hace tres cosas cuando recibe una imagen:
1. Descarga los bytes de imagen desde la API de MiniMax
2. Guarda el archivo localmente en `/root/minimax-agent/outputs/img_*.png`
3. Codifica los bytes en base64

Sin embargo, `cmd_imagine()` en `main.py` solo usaba `result["url"]` (la URL OSS remota) para enviar a Telegram, ignorando completamente `result["local_path"]` que ya contenía la ruta al archivo local.

La sesión `20260407_050523` ya había identificado este problema, pero el fix nunca se aplicó al código de producción — quedó solo en la documentación del pipeline.

## 4. Qué se Corrigió

### Cambio en main.py — cmd_imagine()
**Antes (roto):**
```python
image_url = result.get("url")
if image_url:
    await update.message.reply_photo(photo=image_url)
```
**Después (corregido):**
```python
local_path = result.get("local_path", "")
b64_data = result.get("base64_data", "")

if local_path and os.path.exists(local_path):
    # Enviar archivo binario local — método preferido
    logger.info(f"[cmd_imagine] send_method=local_binary local_path={local_path}")
    with open(local_path, "rb") as f:
        await update.message.reply_photo(photo=f)
elif b64_data:
    # Fallback: decodificar base64 y enviar como binario
    logger.warning(f"[cmd_imagine] send_method=base64_fallback local_path={local_path}")
    img_bytes = base64.b64decode(b64_data)
    await update.message.reply_photo(photo=BytesIO(img_bytes))
elif image_url:
    # Último recurso: enviar URL remota
    logger.warning(f"[cmd_imagine] send_method=remote_url_fallback local_path={local_path}")
    await update.message.reply_photo(photo=image_url)
else:
    await update.message.reply_text("❌ No se recibió URL de imagen.")
```

### Cambio en brain.py — execute_tool_call para generate_image
Ahora incluye `base64_data` en la respuesta JSON del tool call, y añade una aserción que verifica que `base64_data` no esté vacío cuando `local_path` existe:

```python
# brain.py líneas 599-600
if not error and result_dict.get("local_path"):
    assert result_dict.get("base64_data"), \
        f"generate_image returned local_path but no base64_data — download may have failed silently"
```

### Validación de startup — _validate_generate_image_response()
Nueva función en `main.py` que se ejecuta al iniciar el bot. Hace una llamada real a `generate_image()` y verifica que:
- Si `local_path` existe → `base64_data` no debe estar vacío
- Si está vacío → lanza `AssertionError` con log CRITICAL (`IMAGE_SEND_GAP`)

Esto asegura que si el download falla silenciosamente en el futuro, el bot lo detecta al arrancar y no funciona en un estado degradado.

## 5. Mejoras Estructurales Añadidas

### 1. Envío multinivel con fallback
El fix implementa 3 niveles de envío de imágenes:
- **Nivel 1 (preferido):** Archivo local binario (`open(local_path, 'rb')`)
- **Nivel 2 (fallback):** Decodificación base64 → `BytesIO`
- **Nivel 3 (último recurso):** URL remota OSS

Esto significa que incluso si el archivo local se pierde o el sistema de archivos falla, el bot puede reconstruir la imagen desde base64 en memoria.

### 2. Observabilidad completa
Cada método de envío queda registrado con un log estructurado:
- `[cmd_imagine] send_method=local_binary` — archivo local enviado exitosamente
- `[cmd_imagine] send_method=base64_fallback` — se usó decodificación base64
- `[cmd_imagine] send_method=remote_url_fallback` — se usó URL remota como último recurso

### 3. Validación de startup
`_validate_generate_image_response()` detecta fallas silenciosas de download antes de que el bot entre en producción. Si `base64_data` está vacío cuando `local_path` existe, el sistema lanza un `IMAGE_SEND_GAP` CRITICAL y falla inmediatamente.
