# Fix Report — Image generation via tool call sends URL not binary photo

## 1. Problema Reportado

Fernando envió el mensaje: *"Que paso? No que me ibas a mandar la imagen en base o en binario y no en un link que no puedo abrir"*

El bot de Telegram (MiniMax-Agent) estaba enviando un enlace URL de OSS (por ejemplo, `http://hailuo-image-algeng-data-us.oss-us-east-1.aliyuncs.com/...`) en lugar de enviar la imagen como archivo binario directamente en el chat. El usuario esperaba recibir la foto directamente, no un link a la imagen.

Contexto de la conversación:
1. Fernando pidió que el bot generara una imagen de "Visión" (su nuevo nombre para el bot)
2. El bot generó la imagen exitosamente usando `generate_image()` vía tool-calling de MiniMax
3. En lugar de enviar la imagen en binario, el bot envió el enlace OSS como texto
4. Fernando se frustró porque no podía ver la imagen desde ese enlace

## 3. Causa Raíz

### Versión accessible para Fernando

**Qué pasaba:** Cuando el bot usaba `generate_image()` como tool-call durante una conversación de chat, la función `execute_tool_call()` en `brain.py` recibía el resultado con la ruta local del archivo (`local_path`) y los datos en base64 (`base64_data`). Sin embargo, todo esto se convertía a un string JSON que solo el LLM veía. El LLM luego incluía la URL de OSS en su respuesta de texto, y `handle_text()` en `main.py` enviaba ese texto por `reply_text()` — la imagen nunca se enviaba como foto binaria.

### Detalle técnico

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Función:** `execute_tool_call()` (línea ~597) + `chat()` tool-call loop (líneas ~1038-1056)

```
Flujo ANTES del fix:
execute_tool_call("generate_image")
  → image_handler.generate_image() returns {"url", "local_path", "base64_data"}
  → return json.dumps({...})  ← JSON string solo visible para el LLM
  → LLM produce text response con OSS URL
  → handle_text() → reply_text(oss_url)  ← NUNCA se envía como binary photo
```

**Por qué fallaron 3 intentos anteriores:** Las correcciones previas intentaban agregar `_detect_and_send_image()` con regex para encontrar rutas de imagen en el texto del LLM. Pero este enfoque era frágil porque:
1. El regex solo matcheaba `img_[a-f0-9]+\.png` — fallaba con URLs OSS completas
2. El regex se ejecutaba DESPUÉS de que el texto ya se había enviado
3. El `local_path` real estaba disponible en el tool-result pero nunca se usaba

**Confidence del root cause:** 0.95

## 4. Qué se Corrigió

### Estructura del fix: Typed `ToolResult` / `ImagePending` Protocol

Se implementó un protocolo estructurado para propagar los datos de imagen desde `execute_tool_call` hasta `handle_text`:

**Paso 1 — `ToolResult` namedtuple en brain.py:**
```python
@dataclass
class ToolResult:
    result_type: str  # "text" | "image" | "error"
    data: dict        # parsed JSON data or error details
```

**Paso 2 — `execute_tool_call()` ahora retorna `ToolResult`:**
```python
# ANTES: return json.dumps({"url": ..., "local_path": ..., "base64_data": ...})
# AHORA:
return ToolResult(
    result_type="image",
    data={"url": url, "local_path": local_path, "base64_data": base64_data}
)
```

**Paso 3 — `chat()` propaga datos de imagen pendiente:**
- `chat()` ahora retorna `tuple[str, Optional[dict]]` en lugar de solo `str`
- El `dict` contiene `{local_path, base64_data}` cuando `generate_image` fue llamado

**Paso 4 — `simple_chat()` retorna `ImagePending`:**
```python
@dataclass
class ImagePending:
    local_path: str
    base64_data: str
    text: str  # texto del LLM para enviar después de la foto
```

**Paso 5 — `handle_text()` detecta `ImagePending` y envía foto primero:**
```python
if isinstance(response, ImagePending):
    # Enviar foto binaria primero (ruta local > base64 > URL)
    await update.message.reply_photo(photo=open(response.local_path, 'rb'))
    # Luego enviar texto del LLM
    await update.message.reply_text(response.text)
else:
    # Flujo normal de solo texto
    await update.message.reply_text(response)
```

## 5. Mejoras Estructurales Añadidas

### 1. Prevention of Recurrence (Layer 1 — Type System)
El `ToolResult` namedtuple con campo `result_type` hace imposible que `execute_tool_call` retorne un resultado de imagen sin marcar como tal. El tipo lo fuerza.

### 2. Prevention of Recurrence (Layer 2 — Startup Validation)
```python
def _validate_image_result_protocol():
    """Verifica que execute_tool_call retorna ToolResult para generate_image."""
    from agent.brain import ToolResult, ImagePending
    assert hasattr(ToolResult, 'result_type'), "ToolResult missing result_type field"
    assert hasattr(ToolResult, 'data'), "ToolResult missing data field"
    logger.info("[startup] image_result_protocol validated OK")
```
Se ejecuta al cargar el módulo (`brain.py` línea ~1205). Si el protocolo se rompe en el futuro, el bot falla en startup con un log CRITICAL.

### 3. Prevention of Recurrence (Layer 3 — Observability)
Logs estructurados agregados en 3 puntos críticos:
- `[execute_tool_call] generate_image result_type=image local_path=...` (brain.py)
- `[simple_chat] image_pending detected, local_path=...` (brain.py)
- `[handle_text] sending image via reply_photo local_path=...` (main.py)

### 4. Prevention of Recurrence (Layer 4 — Unit Tests)
`tests/test_image_tool_call_protocol.py` con 4 tests que validan el contrato:
- `execute_tool_call` retorna `ToolResult(result_type="image")` para `generate_image`
- `execute_tool_call` retorna `ToolResult(result_type="error")` en rate limit
- `execute_tool_call` retorna `ToolResult(result_type="error")` cuando no disponible
- `simple_chat` retorna `ImagePending` cuando `generate_image` tiene éxito

### 5. Fallback Chain en `handle_text`
Si `local_path` no existe o falla, se prueba con `base64_data` decodificado, y como último recurso la URL OSS. No hay single point of failure.
