## 3. Causa Raíz

### Explicación para Fernando

Cuando el bot pidió a MiniMax que generara una imagen, la API de MiniMax respondió correctamente con HTTP 200 (es decir, no hubo error de red ni error HTTP), pero el cuerpo de la respuesta era `{"choices": null}` — la llave `"choices"` existía en el JSON pero su valor era `null` (no un array, sino literalmente "nada").

El código de `brain.py` hacía esto:

```python
data = await call_minimax(messages, tools=tools)  # ← retornó {"choices": null}
choice = data.get("choices", [{}])[0]              # ← crash aquí
```

La función `.get("choices", [{}])` tiene un segundo argumento `[{}]` que es el valor por defecto que debe usarse cuando la llave `"choices"` NO existe. Pero en este caso, la llave SÍ existe (con valor `null`), así que Python ignora el valor por defecto y regresa `None`. Luego `[0]` sobre `None` causa el `TypeError`.

**Esto no es un bug del código normal** — es un caso extremo donde la API de MiniMax, probablemente por estar saturada o en estado degradado, devolvió una respuesta semánticamente vacía (200 OK pero sin contenido real).

### Detalle técnico

- **Archivo:** `/root/minimax-agent/agent/brain.py`
- **Función:** `chat()` en la línea 984-990
- **Mecanismo exacto:** `data.get("choices", [{}])` retornó `None` porque la llave `"choices"` estaba presente con valor `null` en la respuesta JSON — el default `[{}]` solo se usa cuando la llave está ausente, no cuando vale `None`
- **Evidencia del log:** HTTP 200 → `call_minimax Succeeded on attempt 1` → crash en 3ms — descarta timeout o problema de red
- **Segundo error en la misma sesión:** Mismo patrón `{"choices": null}` en una segunda llamada API, causando que `response_text` nunca fuera asignado

## 4. Qué se Corrigió

Se aplicaron **4 cambios estructurales** a `brain.py`:

### Fix 1 — Guard contra null-choices en `chat()` (línea 984)
```python
data = await call_minimax(messages, tools=tools)
if data is None or not isinstance(data.get("choices"), list):
    logger.error(f"[chat] MiniMax returned null choices — data={data!r}")
    return ("No pude obtener respuesta de MiniMax. Intenta de nuevo.", None)
```
Este es el fix primario: antes de hacer `[0]` sobre el resultado, verifica que `data` no sea `None` y que `choices` sea realmente una lista.

### Fix 2 — Valor de retorno `Optional[dict]` en `call_minimax()`
El tipo de retorno de `call_minimax()` decía `-> dict` pero en realidad podía retornar `None` cuando `response.json()` recibía un body vacío. Se cambió a `-> Optional[dict]` y se agregó una aserción:

```python
result = response.json()
if result is None:
    logger.warning("[call_minimax] response.json() returned None, treating as retryable failure")
    raise ValueError("MiniMax API returned null/empty body")
return result
```

Esto convierte el caso `null body` en un error typed que dispara el retry en lugar de retornar `None` silenciosamente.

### Fix 3 — Tipo de retorno corregido
`call_minimax` ahora dice `-> Optional[dict]` en vez de `-> dict`, reflejando el contrato real de la función.

### Fix 4 — Función de validación `_validate_minimax_response_contract()`
Nueva función al final de `brain.py` que hace una llamada de prueba a MiniMax al iniciar el módulo y verifica que la respuesta tenga `choices` como lista no-null. Si MiniMax está en estado degradado al arrancar, loguea un WARNING en lugar de crashear silenciosamente.

## 5. Mejoras Estructurales Añadidas

1. **Defensa en profundidad de 2 capas:** La primera capa (guard en `chat()`) previene el crash directamente en el punto de uso. La segunda capa (ValueError en `call_minimax()`) convierte el edge case en un error typed que dispara retry, cubriendo cualquier otro lugar que llame a `call_minimax`.

2. **Tipo de contrato explícito:** `Optional[dict]` hace visible en type-checkers que esta función puede retornar `None`. Sin esto, un type checker no detectaría el problema.

3. **Startup health check:** `_validate_minimax_response_contract()` corre automáticamente al importar `brain.py` y detecta si MiniMax está degradado desde el arranque, dando visibilidad temprana.

4. **Suite de tests completa:** Se creó `tests/test_minimax_null_choices.py` con 8 tests cubriendo:
   - `{"choices": None}` → error tuple graceful
   - `None` data → error tuple graceful
   - `response.json()` retorna `None` → `ValueError` disparado (retry)
   - Respuesta válida → comportamiento normal sin cambios
   - Startup validation con respuesta válida → `True`
   - Startup validation con `choices=null` → `False`
   - Startup validation con respuesta `None` → `False`
   - Startup validation con excepción → `False`
