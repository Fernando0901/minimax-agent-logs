# Fix Report — TypeError en tool-call loop de brain.py (MiniMax null body)

---

## 1. Problema Reportado

El bot de Telegram (MiniMax-Agent) registró un error crítico que detuvo el procesamiento de mensajes entrantes:

```
TypeError: 'NoneType' object is not subscriptable
  en brain.py, línea 999: choice = data.get("choices", [{}])[0]
```

**Mensaje que lo conmemó:** "genera una imagen del proyecto visión como tú lo imaginas" (usuario ID 8288612046, 14:00:10 UTC).

El error occurred dentro del tool-call loop de la función `chat()` en `brain.py`, específicamente en la iteración 2+ del loop que maneja respuestas de la API de MiniMax que incluyen tool calls.

## 3. Causa Raíz

### Causa técnica

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Función:** `chat()` (líneas 988-1117)
**Mecanismo:** La guarda nula (`if data is None or not isinstance(data.get("choices"), list)`) estaba posicionada SOLO después del primer llamado a `call_minimax` (línea 988), antes de entrar al tool-call loop. Los llamados subsecuentes a `call_minimax` dentro del loop (líneas 1051 y 1113) NO tenían guarda nula. Si la API de MiniMax regresaba HTTP 200 con cuerpo null/empty en cualquier iteración 2+ del loop, `data` se convertía en `None` y la línea 999 (`choice = data.get("choices", [{}])[0]`) lanzaba `TypeError` porque `None.get()` devuelve `None` y `None[0]` no es subscriptable.

### Por qué falló la guarda anterior

La guarda añadir en la sesión `20260407_133027` solo protegió el primer llamado a `call_minimax`. El tool-call loop (añadido después para soportar encadenamiento de herramientas) nunca fue protegido con la misma guarda en cada iteración.

### Evidencia
- Línea 999 es donde ocurre el error — dentro del loop `for _ in range(max_iterations)`, no antes de entrar al loop
- El error ocurre en iteración 2+ (no en la primera llamada)
- `call_minimax` puede devolver `None` sin lanzar excepción cuando `response.json()` retorna `None` y el check `if result is None` no dispara por alguna condición de carrera

---

## 4. Qué se Corrigió

### Cambio 1 — Guarda en top del loop (línea ~1001)
```python
# Guard before every data access in loop (catches None from any call_minimax call in loop)
if data is None or not isinstance(data.get("choices"), list):
    logger.error(f"[chat] MiniMax returned null/empty body at loop start — data={data!r}")
    return ("No pude obtener respuesta de MiniMax. Intenta de nuevo.", None)
```
Esta guarda se ejecuta ANTES de cada acceso a `data` dentro del loop, detectando `None` de cualquier llamado anterior a `call_minimax` dentro del loop.

### Cambio 2 — Guarda después del text-tool-call call (línea ~1059)
Después del workaround para tool calls basados en texto, se agregó la misma guarda antes del `continue`:
```python
if data is None or not isinstance(data.get("choices"), list):
    logger.error(f"[chat] MiniMax returned null/empty body after text-tool-call — data={data!r}")
    return ("No pude obtener respuesta de MiniMax. Intenta de nuevo.", None)
```

### Cambio 3 — Guarda después del tool-results call (línea ~1126)
Después de procesar resultados de herramientas y llamar a `call_minimax` con `tools=tools`:
```python
if data is None or not isinstance(data.get("choices"), list):
    logger.error(f"[chat] MiniMax returned null/empty body in loop iteration — data={data!r}")
    return ("No pude obtener respuesta de MiniMax. Intenta de nuevo.", None)
```

### Cambio 4 — Try/except en simple_chat() (línea ~1156)
```python
try:
    response_text, pending_info = await chat(...)
except (ValueError, TypeError) as exc:
    logger.warning(f"[simple_chat] chat() raised {type(exc).__name__}, returning graceful error")
    return "No pude obtener respuesta de MiniMax. Intenta de nuevo."
```
Esto evita que excepciones de `chat()` se propaguen hasta `handle_text()` causando el UnboundLocalError en cascada.

---

## 5. Mejoras Estructurales Añadidas

1. **Guarda en 3 puntos dentro del loop** — enfoque "belt-and-suspenders": la guarda ahora existe al inicio del loop, después del call_minimax de text-tool-call, y después del call_minimax de tool-results. Cualquier `data = None` de cualquier llamado será detectado antes de acceder a `data.get("choices")[0]`.

2. **Try/except en simple_chat()** — capa de defensa adicional: si por cualquier razón la guarda falla y `chat()` lanza `ValueError` o `TypeError`, `simple_chat()` lo captura y retorna un mensaje de error graceful en lugar de propagar la excepción hasta `handle_text()`.

3. **Logging estructurado** — cada guarda nula registra el estado de `data` completo para facilitar debugging en producción.

4. **Suite de tests** — `tests/test_minimax_null_body_loop.py` con 5 tests cubriendo todos los escenarios de null body:
   - `test_chat_loop_guard_first_iteration`
   - `test_chat_loop_guard_returns_empty_dict`
   - `test_chat_null_choices_in_loop`
   - `test_simple_chat_catches_chat_valueerror`
   - `test_simple_chat_catches_chat_typeerror`

## 7. Issues Pre-existentes Encontrados

**Issue P1 (Resuelto):** La guarda nula solo protegía el primer `call_minimax`, no las iteraciones del tool-call loop. → RESUELTO con los 3 nuevos guards.

**Issue P2 (Resuelto):** `simple_chat()` no tenía try/except alrededor de `await chat()`. → RESUELTO con try/except en línea 1156.

**Issue P3 (Pendiente):** El mismo error `TypeError: 'NoneType' object is not subscriptable` se repitió en la sesión `20260407_133027` (agregó guarda solo al primer call) y ahora en `20260407_140628` (agregó guards a loop iterations). La guarda de la sesión anterior fue incompleta porque no cubrió las iteraciones 2+ del loop. Esta sesión completó el fix.

**Otros errores en audit log** — Los errores de UnboundLocalError en `response_text` (13:31 y 14:02) son errores en cascada del mismo problema: cuando `chat()` lanza, `response_text` nunca se asigna, causando el error secundario en `log_outgoing()`. El try/except añadido en `simple_chat()` corta esta cascada en la raíz.
