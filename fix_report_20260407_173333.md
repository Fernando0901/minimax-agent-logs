# Fix Report — MiniMax-Agent: MiniMax Embeds OSS Image URL in Text Response Instead of Sending as Binary

---

## 1. Problema Reportado

Fernando utilizó el comando `/imagine` para generar una imagen. El sistema generó la imagen correctamente, pero cuando la respuesta del LLM (MiniMax M2.7) incluía la URL de OSS (Object Storage Service) directamente en el texto de respuesta — en lugar de retornar solo el `local_path` o `base64_data` — el bot envió la URL como texto plano en Telegram. Fernando recibió un link que no puede abrir como archivo binario, en vez de la imagen directamente.

Además, no había forma de reenviar una imagen previamente generada porque el sistema no almacenaba el binary de la imagen — solo guardaba la URL de OSS en el historial de conversación.

Los errores documentados en los logs fueron:
- `TypeError: 'NoneType' object is not subscriptable` (13:30:27 y 14:00:10)
- `UnboundLocalError: cannot access local variable 'response_text'` (13:31:13 y 14:02:06)

Estos cuatro errores tuvieron la misma causa raíz: `brain.py:simple_chat` retornaba `None` cuando la API de MiniMax regresaba `{"choices": null}` (estado degradado), y no había fallback en ese caso. Estos errores ya habían sido resueltos en sesiones anteriores del mismo día (20260407_133027, 20260407_134700, 20260407_140628, 20260407_143451). El problema reportado en esta sesión es específicamente sobre la URL de imagen en texto.

## 3. Causa Raíz

### Causa Raíz Técnica

**Archivo:** `/root/minimax-agent/agent/main.py` + `/root/minimax-agent/agent/brain.py`

**Mecanismo:** Cuando MiniMax M2.7 genera una imagen via la herramienta `generate_image`, a veces incluye la URL de OSS del archivo resultante directamente en el texto de respuesta del LLM. Por ejemplo:

```
¡Aquí está tu imagen! http://hailuo-image-algeng-data-us.oss-east-1.aliyuncs.com/image_inference_output/talkie/prod/img/2026-04-07/abc.png Aquí está el enlace.
```

El flujo existente de `ImagePending` (ToolResult namedtuple → `reply_photo` binario) solo cubría el caso donde la imagen se enviaba como binary attachment. No había código que detectara, descargara y reenviara como binary cuando la URL aparecía embebida en texto libre.

Adicionalmente, el sistema no contaba con persistencia de la imagen binaria — solo almacenaba la URL de OSS en el historial de conversación, por lo que no era posible reenviar la imagen sin regenerarla.

### Causa Raíz Secundaria (errores de logs)

Los cuatro eventos de error en los logs (`TypeError: 'NoneType' object is not subscriptable` y `UnboundLocalError: cannot access local variable 'response_text'`) fueron causados por `simple_chat()` retornando `None` cuando la API de MiniMax regresaba `{"choices": null}` en estado degradado. La guarda en `chat()` línea 992 (`if data is None or not isinstance(data.get("choices"), list)`) debería atrapar este caso, pero había una condición de carrera con bytecode stale del contenedor. Estos errores ya fueron resueltos en las sesiones 20260407_133027, 20260407_140628 y 20260407_143451.

## 4. Qué se Corrigió

### Fix 1 — URL Stripping en handle_text (`main.py`)
Nueva función `_extract_and_strip_image_url(text: str) -> tuple[str, str | None]` que:
1. Compila regex para patrones OSS: `r'(https?://[^\s]*?(?:hailuo-image[^\s]*?\.png|image_inference_output[^\s]*?\.(?:png|jpg|jpeg)))'`
2. Detecta si el texto contiene una URL de imagen OSS
3. Si encuentra URL: descarga la imagen via httpx (timeout 30s), guarda a archivo temporal, retorna `(texto_sin_URL, ruta_local)`
4. Si la descarga falla: retorna `(texto_original, None)` — preserva comportamiento existente como fallback seguro
5. Si no encuentra URL: retorna `(texto_original, None)`

Integrada en `handle_text` antes del check de `ImagePending`, para que si MiniMax pone la URL en texto Y también envía ImagePending, la URL se remueva del texto antes de paginar.

### Fix 2 — Last Image Cache en brain.py
Nuevo módulo `_last_image_cache = {}` dict + función `get_last_image()` que retorna una copia del cache (o `None` si vacío). Cuando `execute_tool_call("generate_image")` se ejecuta exitosamente, el resultado se almacena en el cache con las llaves: `url`, `local_path`, `base64_data`.

### Fix 3 — Comando /image para reenvío explícito
Nuevo handler `cmd_image` que:
1. Llama `brain.get_last_image()`
2. Si el cache tiene imagen: envía como binary via `reply_photo` (local_path preferido, luego base64, luego URL)
3. Si el cache está vacío: responde "No hay imagen reciente para reenviar."
4. Registrar con `app.add_handler(CommandHandler("image", cmd_image))`

### Fix 4 — Validaciones de Startup
- `_validate_image_url_stripping()`: prueba que el regex detecta URLs hailuo y que las stripping works correctly. Falla con CRITICAL log si el regex se rompe.
- `_validate_last_image_cache()`: prueba que el cache inicia vacío y `get_last_image()` retorna `None`. Falla con CRITICAL log si el cache no funciona.

Ambas validaciones corren al iniciar el contenedor y sus resultados aparecen en los logs de Docker.

## 5. Mejoras Estructurales Añadidas

1. **Capa de validación al startup**: Dos nuevas validaciones que fallan rápido con CRITICAL log si se rompen, detectando regresiones antes de que produzcan errores en producción.

2. **Cache de imagen persistente en memoria**: `_last_image_cache` permite reenvío sin regeneración, resolviendo la limitación de no poder recuperar una imagen previamente generada.

3. **Logs estructurados para observabilidad**:
   - `[handle_text] stripping image URL from text, url={url}` — confirma que stripping ocurrió
   - `[handle_text] no image URL found in text` — confirma camino normal
   - `[execute_tool_call] cached generate_image result, local_path={path}` — confirma caching
   - `[handle_text] failed to send stripped URL image: {img_err}` — alerta cuando download falla

4. **Test suite nueva**: `tests/test_image_resend_and_stripping.py` con 7 tests cubriendo: detección de URL, stripping correcto, caso sin URL, cache populate, cache retrieval, comando /image con cache, comando /image sin cache.
