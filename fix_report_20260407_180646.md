## 3. Causa Raíz

### Causa Raíz Técnica — Imagen no enviada como binario

**Archivo:** `/root/minimax-agent/agent/main.py` (función `handle_text`, branch de URL-stripping)
**Mecanismo:** Cuando MiniMax embebe la URL de OSS de la imagen directamente en el texto de respuesta (en lugar de usar el path de `ImagePending` vía tool_calls), la función `_extract_and_strip_image_url()` detectaba y removía correctamente la URL del texto a mostrar. SIN EMBARGO, la imagen generada estaba cacheada en `_last_image_cache` (durante `execute_tool_call("generate_image")`), y `handle_text` nunca la recuperaba para enviarla como binario. Fernando recibía el texto limpio, pero SIN la imagen.

El flujo original era:
1. LLM genera imagen → `execute_tool_call("generate_image")` guarda `base64_data` en `_last_image_cache`
2. LLM incluye URL de OSS en texto → `handle_text` detecta y strippea la URL del texto
3. **PROBLEMA:** Se enviaba solo el texto limpio, la imagen cacheada nunca se recuperaba

### Causa Raíz Técnica — Resumen de auditoría falso

**Archivo:** `/root/minimax-agent/agent/self_improve.py` (función `queue_fix_task`)
**Mecanismo:** `queue_fix_task` generaba el resumen de auditoría para el task de self-fix usando `today_events("error", limit=10)` para poblar `error_lines`. Si `today_events()` retornaba una lista vacía (por una excepción silenciada o path de archivo incorrecto), `error_lines` quedaba vacío, y el resumen final decía "No errors detected" aunque `today_summary()` retornaba `{'error': 4}`.

## 4. Qué se Corrigió

### Fix 1 — main.py: envío de imagen en caché antes de fallback URL

**Ubicación:** `main.py` líneas ~886-899, función `handle_text`, branch de URL-stripping

**Antes:**
```python
# Solo strippeaba y caía a descarga por URL
stripped_url = _extract_and_strip_image_url(response_text)
if stripped_url:
    response_text = response_text.replace(stripped_url, "").strip()
    # fallback: descarga por URL...
```

**Después:**
```python
# Primero intenta imagen cacheada, fallback a URL
stripped_url = _extract_and_strip_image_url(response_text)
if stripped_url:
    cached = get_last_image()
    if cached and cached.get("base64_data"):
        img_bytes = base64.b64decode(cached["base64_data"])
        await update.message.reply_photo(photo=BytesIO(img_bytes))
        logger.info(f"[handle_text] cached image sent ({len(cached['base64_data'])} bytes)")
    elif stripped_url:
        logger.critical(f"[handle_text] OSS URL stripped but _last_image_cache EMPTY — image NOT sent! url={stripped_url}")
    response_text = response_text.replace(stripped_url, "").strip()
    # fallback: descarga por URL...
```

### Fix 2 — self_improve.py: fallback para error_lines en queue_fix_task

**Ubicación:** `self_improve.py` líneas ~507-532, función `queue_fix_task`

**Cambio:** Envolvió la llamada a `today_events("error", limit=10)` en try/except. Si `today_events` retorna lista vacía pero `today_summary().get('error', 0) > 0`, cae a lectura directa del archivo JSONL de auditoría filtrando por `type:"error"`.

### Fix 3 — Contenedor rebuildado

Ejecutó `docker-compose -f /root/minimax-agent/docker-compose.yml up --build -d`. El contenedor `minimax-agent` fue recreado, Started, y verificado saludable (`Up About a minute (healthy)`).

## Resumen Ejecutivo

Fernando reportó que cuando pidió una imagen en lenguaje natural, el bot le envió un link de OSS en lugar de la imagen como archivo binario. El pipeline identificó que la causa raíz era que `_extract_and_strip_image_url()` removía la URL del texto pero nunca recuperaba la imagen cacheada en `_last_image_cache` para enviarla por `reply_photo`. Adicionalmente, el sistema de auditoría generaba resúmenes falsos ("No errors detected") cuando había errores reales en los logs.

La solución implementada hace que `handle_text` intente PRIMERO enviar la imagen cacheada (via `get_last_image()`) y solo caiga al fallback de descarga por URL si la caché está vacía, emitiendo un log CRITICAL si esto ocurre. También se agregó un fallback en `queue_fix_task` para asegurar que `error_lines` siempre se pueble correctamente del archivo de auditoría.

El contenedor fue rebuildado y está saludable. 6 tests passando. Ningún error nuevo introducido.
