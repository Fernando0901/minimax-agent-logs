# Fix Report — odoo_bridge.py: errores de conexión al microservicio Odoo

---

## 1. Problema Reportado

Fernando reportó que el microservicio `asistente_odoo` estaba teniendo problemas para responder desde el webhook, devolviendo el mensaje: **"Estoy teniendo problemas técnicos. Por favor intenta de nuevo en un momento. 🙏"**

Este mensaje de error es generado por el propio microservicio de Odoo cuando no puede procesar una solicitud — ya sea porque el servicio de Odoo está caído, porque la conexión a la base de datos falló temporalmente, o porque el servidor de Odoo no responde dentro del tiempo esperado.

## 3. Causa Raíz

### Causa técnica

**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Función:** Las 12 funciones del módulo (`search`, `get_product_price`, `get_product_bom`, `create_contact`, `create_opportunity`, `update_opportunity`, `schedule_activity`, `create_quotation`, `get_overdue_activities`, `batch_price`, `health_check`, `agent_chat`)
**Líneas:** 1-144 (todo el módulo)

**Mecanismo:** Cada función en `odoo_bridge.py` crea un `httpx.AsyncClient` y llama `await c.post()` / `await c.get()` sin ningún bloque `try/except`. Cuando el microservicio en `odoo_python_agent:8000` estaba inalcanzable, la conexión rechazada lanzaba `httpx.ConnectError` como excepción no capturada. Cuando expiraba el timeout, `httpx.Timeout` era lanzado. Cuando retornaba un estado no-2xx, `httpx.HTTPStatusError` era lanzado. Ninguna de estas excepciones era capturada, por lo que se propagaban por la cadena de llamadas: `execute_tool_call("odoo_agent")` → `chat()` → `simple_chat()`.

Adicionalmente, cuando el microservicio retornaba una respuesta HTTP 200 válida con JSON de error (`{"status":"error","response":"Estoy teniendo problemas técnicos..."}`), `execute_tool_call` extraía el mensaje de error y lo retornaba como texto plano, causando que el usuario viera el mensaje de error técnicas.

**También se encontró causa secundaria:** El fix aplicado en la sesión `20260407_195554` añadió `logger = logging.getLogger("audit")` en `self_improve.py`, pero si el contenedor Docker no fue reconstruido (`docker-compose up --build`) después de ese fix, el bytecode stale seguía corriendo sin el binding del logger, causando `NameError: name 'logger' is not defined` en la ruta de fallback de `execute_self_fix`.

**Por qué el código permitió el problema:** El diseño original asumía que el microservicio siempre estaría saludable y siempre retornaría respuestas HTTP 200 válidas. No había manejo defensivo de errores porque el desarrollador no consideró fallos transient de red, caídas de conexión a Odoo, o reinicios del microservicio.

## 4. Qué se Corrigió

### Cambio 1 — `agent/odoo_bridge.py` (principal)
Se añadió `try/except httpx.HTTPError` alrededor de cada llamada httpx en las 12 funciones del módulo. Cada función ahora retorna `{"error": str(e)}` en caso de fallo en lugar de propagar la excepción. Específicamente para `agent_chat()`, también se detectan respuestas `{"status":"error",...}` y se extrae el mensaje de error interno para retornarlo de manera controlada.

### Cambio 2 — `agent/brain.py` (secundario)
Se amplió el manejador de excepciones en `simple_chat()` de `except (ValueError, TypeError)` a `except Exception`, asegurando que cualquier excepción de `chat()` sea capturada y retorne un string de error graceful en lugar de propagarse sin manejar.

### Cambio 3 — `agent/odoo_bridge.py` (nuevo)
Se añadió la función `_validate_odoo_bridge_endpoints()` como validación de startup. Esta función hace ping a cada endpoint del microservicio con un timeout de 5 segundos, cuenta los fallos, y emite un log CRITICAL si más del 50% de los endpoints fallan. Esto permite que el bot inicie incluso si el microservicio está caído, pero proporciona visibilidad inmediata del estado de salud.

### Cambio 4 — `agent/main.py` (nuevo)
Se añadió la llamada a `_validate_odoo_bridge_endpoints()` en el startup del bot, después de la validación de `generate_image`. Esto asegura que la validación de endpoints del microservicio se ejecute automáticamente cuando el contenedor inicia.

### Cambio 5 — `tests/test_odoo_bridge_error_handling.py` (nuevo)
Se creó una suite de 6 tests cubriendo: manejo de `ConnectError`, manejo de `Timeout`, detección de respuesta de error JSON, manejo de `HTTPStatusError`, y logging de la validación de startup.

---

## 5. Mejoras Estructurales Añadidas

1. **Manejo de errores en capa de puente:** Todas las 12 funciones en `odoo_bridge.py` ahora tienen `try/except` alrededor de las llamadas httpx. Cada función retorna un dict controlado (nunca lanza excepción). La traducción de excepción-a-error-dict ocurre en la capa del puente, así que los llamadores siempre reciben un dict que pueden inspeccionar.

2. **Validación de startup:** La función `_validate_odoo_bridge_endpoints()` corre automáticamente cuando el contenedor inicia, proporcionando visibilidad inmediata del estado de salud del microservicio. Si más del 50% de los endpoints fallan, se emite un log CRITICAL.

3. **Logs estructurados:** Se añadieron logs estructurados en `odoo_bridge.py` para éxito y fracaso de cada llamada: `logger.warning(f"[odoo_bridge] {func_name} failed: {type(e).__name__}: {e}")` en excepciones, `logger.info(f"[odoo_bridge] {func_name} succeeded")` en éxito.

4. **Prevención de regresión por bytecode stale:** La excepción en `simple_chat()` ahora captura `Exception` en lugar de solo `(ValueError, TypeError)`, previniendo la propagación de errores no manejados.
