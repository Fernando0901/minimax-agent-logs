# Fix Report — Bot hallucinations: "Gemini 3 Pro" y "Fix Pipeline" inventados

## 1. Problema Reportado

Fernando reportó que el bot de Telegram estaba experimentando alucinaciones severas durante una conversación:

**Síntomas observados:**
1. El bot afirmó incorrectamente estar usando "Gemini 3 Pro" cuando en realidad usa MiniMax M2.7
2. El bot inventó un "Fix Pipeline" workflow que no existe en n8n
3. El bot afirmó poder "ver logs" de sesiones pasadas del pipeline sin usar herramientas
4. Tiempos de respuesta anómalos: 15-28 segundos normalmente, pero una respuesta de solo 8.4ms
5. El bot no usó herramientas disponibles (`read_file`, `n8n_list_workflows`) correctamente para responder preguntas sobre el pipeline

La conversación exacta registrada en los logs mostró:
- Bot afirmó ver "logs de la última sesión pipeline del 04 de abril"
- Bot inventó nombres de workflows como "Fix Pipeline", "Architectural Bot Supervisor-Worker (Gemini 3 Pro)"
- Fernando tuvo que corregir al bot múltiples veces: "no estoy usando Gémini 3 pro, sino minimax"

## 3. Causa Raíz

### Causa Raíz Técnicos (inglés — para referencia del equipo)

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Función:** `build_system_prompt()` — líneas 229-275

**Mecanismo identificado:** El prompt del sistema generado por `build_system_prompt()` no describe la arquitectura del pipeline. Específicamente:

1. No menciona que "el pipeline" es un script Python CLI (`orchestrator.py`) corriendo como proceso separado del bot de Telegram
2. No hay instrucción para usar la herramienta `read_file` cuando se pregunta sobre sesiones o estado del pipeline
3. No hay aplicación de identidad del modelo — el modelo puede reclamar ser "Gemini 3 Pro" u otro

Cuando se le preguntó al modelo sobre el estado del pipeline, cayó en su conocimiento interno de entrenamiento sobre pipelines de CI/CD (que típicamente SÍ tienen workflows de "fix/apply"), en lugar de usar las herramientas disponibles (`read_file`, `n8n_list_workflows`).

**Evidencia:**
- El bot TIENE la herramienta `read_file` (líneas 668-681) y `n8n_list_workflows` — las herramientas existen y funcionan
- El bot sÍ llamó `n8n_list_workflows` en la conversación (respondió con Asistente_Odoo y Architectural Bot Supervisor-Worker)
- El bot afirmó ver "logs del 04 de abril" sin ninguna llamada a `read_file` en los audit logs — el modelo usó conocimiento interno en lugar de herramientas
- El bot inventó el nombre de workflow "Fix Pipeline" — fuga de conocimiento de entrenamiento
- El prompt del sistema (línea 234) dice "You are the personal AI agent" sin descripción de qué es "el pipeline"

**Confidence:** 0.75

### Explicación para Fernando (español)

El problema era que el bot de MiniMax no tenía en su "instrucciones base" una descripción clara de cómo estaba estructurado el sistema de correcciones automáticas. Cuando Fernando le preguntó sobre el pipeline, el modelo de MiniMax intentó responder usando lo que "sabe de entrenamiento" sobre cómo funcionan los pipelines de IA — que típicamente incluyen workflows con nombres como "Fix Pipeline" y asumen que el bot puede "ver logs".

El modelo tenía las herramientas correctas (`read_file` para leer archivos, `n8n_list_workflows` para ver workflows) pero el prompt no le decía que las usara para este propósito. Así que usó su conocimiento interno en lugar de las herramientas, inventando información.

## 4. Qué se Corrigió

Se implementaron los siguientes cambios en `brain.py`:

**1. Nueva sección `## PIPELINE ARCHITECTURE` en `build_system_prompt()`:**
- Describe que el pipeline es un script Python CLI en `/root/minimax-agent/pipeline/orchestrator.py`
- Indica que los archivos de sesión están en `/root/minimax-agent/pipeline/sessions/{timestamp}/`
- Instruye explícitamente usar la herramienta `read_file` cuando se pregunta sobre sesiones, logs o estado del pipeline
- Regla anti-hallucinación: nunca inventar nombres de workflows, nunca reclamar haber ejecutado un workflow sin usar una llamada a herramienta

**2. Nueva sección `## MODEL IDENTITY` en `build_system_prompt()`:**
- Declara explícitamente: "You are powered by MiniMax M2.7"
- Regla: "Never claim to be Gemini, Claude, GPT, or any other AI model"
- Si alguien pregunta qué modelo es: responder "MiniMax M2.7"

**3. Nueva función `_validate_model_identity()` (startup check):**
- Verifica al cargar el módulo que el modelo usado en llamadas API coincide con `MINIMAX_MODEL` env var (default: `MiniMax-M2.7`)
- Si hay discrepancia: lanza `AssertionError` + log CRITICAL → el contenedor falla al iniciar (fail-fast)

**4. Nueva función `_validate_pipeline_architecture_in_prompt()` (startup check):**
- Verifica que el prompt del sistema contenga los tokens requeridos: `read_file`, `pipeline`, `sessions`, `MiniMax-M2.7`
- Si falta algo: lanza `AssertionError` + log CRITICAL → fail-fast

**5. Archivo de tests creado:** `tests/test_pipeline_architecture.py`
- 8 tests cubriendo: identidad del modelo en prompt, arquitectura del pipeline, instrucción de usar `read_file`, cláusula anti-hallucinación, validaciones de startup

**6. `agent_memory.md` actualizado:**
- Memoria persistente ahora incluye descripción de la arquitectura del pipeline
- Asegura que incluso si el prompt se modifica incorrectamente en el futuro, el contexto de memoria incluirá la arquitectura
