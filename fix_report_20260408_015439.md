## 3. Causa Raíz

### Descripción para Fernando

El problema estaba en cómo el bot lee archivos del sistema. Cuando tú le pedías al bot que leyera un archivo (por ejemplo, para entender el pipeline o revisar logs), el bot intentaba leer el archivo usando operaciones de disco **dentro de una función que se ejecuta de forma asíncrona**.

En términos simples: el sistema de archivos del contenedor Docker a veces se "congela" brevemente (especialmente con el sistema overlay de Docker), y como el bot estaba esperando la lectura del archivo **sin poder hacer nada más**, el modelo de IA (MiniMax) esperaba entre 20 y 30 segundos sin recibir respuesta, y como no recibía ningún resultado, **inventaba una respuesta** ("La respuesta tomó demasiado tiempo...").

### Detalle técnico

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Función:** `execute_tool_call()` — específicamente el bloque que maneja la herramienta `read_file` (líneas ~680-693)
**Código problemático:**
```python
# ANTES (bloqueante —不允许asar el control al event loop)
async def execute_tool_call(...):
    if tool_name == "read_file":
        if not os.path.exists(path):          # ← operación síncrona bloqueante
            return json.dumps({"error": ...})
        with open(path, 'r') as f:            # ← operación síncrona bloqueante
            content = f.read(10000)
        return json.dumps({"content": content})
```

Cuando `os.path.exists()` o `open().read()` se bloquean (filesystem stall en overlay de Docker), toda la coroutine `async` se congela. El event loop de asyncio no puede yielding, MiniMax espera indefinidamente, y cuando MiniMax agota su propio timeout interno (~20-30s), genera una respuesta fabricada.

## 4. Qué se Corrigió

### Cambio principal: Wrapper async con timeout

Se reemplazó el handler inline de `read_file` en `execute_tool_call` por una función async nueva con protección de timeout:

```python
# DESPUÉS (no bloqueante — timeout de 15s devuelve error a MiniMax antes de que se agote)
_READ_FILE_TIMEOUT = 15.0

async def _read_file_handler_async(path: str) -> str:
    allowed_dirs = ["/root/minimax-agent", "/opt", "/root"]
    if not any(path.startswith(d) for d in allowed_dirs):
        return json.dumps({"error": "Path not in allowed directories"})
    if not os.path.exists(path):
        return json.dumps({"error": f"File not found: {path}"})
    try:
        with open(path, 'r') as f:
            content = f.read(10000)
        return json.dumps({"content": content, "path": path})
    except Exception as e:
        return json.dumps({"error": str(e)})

# En execute_tool_call, el bloque read_file ahora es:
if tool_name == "read_file":
    return await asyncio.wait_for(
        _read_file_handler_async(path),
        timeout=_READ_FILE_TIMEOUT
    )
```

El timeout de 15 segundos es **más corto** que el timeout interno de MiniMax (~20-30s), garantizando que Python devuelva un error JSON a MiniMax **antes** de que el modelo agote su propio timeout. Así, MiniMax puede manejar el error gracefulmente en lugar de fabricar una respuesta.

### Consolidación de código duplicado

Existían dos implementaciones de `read_file_handler`: un stub `_read_file_handler` en líneas 144-157 (registrado en `_TOOL_REGISTRY`) y un handler inline en líneas 680-693 de `execute_tool_call`. Fueron consolidadas en una sola función `_read_file_handler_async` con timeout.

### Validación de startup

Se添加aron dos verificaciones al iniciar el contenedor:
- `_validate_read_file_timeout()` — usa AST para verificar que `asyncio.wait_for` está presente en `execute_tool_call`
- `_validate_single_read_file_handler()` — cuenta occurrences de `os.path.exists` y `open(` para asegurar que no hay handlers duplicados inline

Si `asyncio.wait_for` falta, el contenedor falla con `CRITICAL: read_file timeout protection missing` y no inicia — detección inmediata.

## Resumen Ejecutivo

**Problema:** El bot fabricaba respuestas de timeout ("La respuesta tomó demasiado tiempo") en lugar de leer archivos del sistema, porque las operaciones de lectura de archivos eran bloqueantes síncronas dentro de una función asíncrona.

**Solución:** Wrapper async con `asyncio.wait_for(..., timeout=15.0)` — ahora si una lectura de archivo se congela más de 15 segundos, Python devuelve un error JSON estructurado a MiniMax, que lo maneja correctamente en lugar de fabricar una respuesta.
