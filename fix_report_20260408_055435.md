# Fix Report — bot hallucinating system state, claiming wrong AI model, and fabricating pipeline workflow names

---

## 1. Problema Reportado

Fernando reportó que el bot de Telegram estaba con "alucinaciones graves":

- **Claim "Gemini 3 Pro"**: El bot se identificó como "Gemini 3 Pro" en lugar de MiniMax-M2.7, el modelo que realmente usa.
- **Workflow inexistente**: El bot inventó un "Fix Pipeline" en n8n que no existe — lo buscó en la lista de workflows y no lo encontró, pero siguió insistiendo en que existía.
- **Archivo no encontrado谎言**: Cuando Fernando preguntó sobre archivos del pipeline, el bot respondió "The file doesn't exist at that path" — una fabricación completa.
- **Timeouts en preguntas complejas**: El bot respondió "La respuesta tomó demasiado tiempo. Intenta una pregunta más simple." en lugar de usar las herramientas disponibles.
- **Sin uso de herramientas**: No usó `read_file` para leer archivos del sistema cuando Fernando preguntó sobre el pipeline — en cambio, entrenó con conocimiento interno y alucinoló.

La conversación mostró el patrón completo: el bot no podía acceder al filesystem, no entendía que el pipeline vive dentro del contenedor minimax-agent (no en n8n), y en lugar de admitir que no tenía acceso, inventó respuestas.

## 3. Causa Raíz

### Causa técnica principal

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Función:** `execute_tool_call()` — bloque `except asyncio.TimeoutError` del handler `read_file`
**Línea:** ~700

El código del handler de timeout de `read_file` usaba `_log.warning(...)` pero `_log` no estaba definido en el scope de `execute_tool_call()`. El logger a nivel de módulo se llama `logger` (línea 36), no `_log`. Cuando el filesystem de Docker overlay se quedaba colgado y disparaba el timeout de 15 segundos del `asyncio.wait_for`, Python levantaba `NameError: name '_log' is not defined` en la línea 700. Este `NameError` se propagaba hacia arriba por el loop de herramientas, dejando al bot en un estado inconsistente. El bot, sin poder obtener una respuesta válida de `read_file`, fabricaba "File not found" en lugar de manejar el error gracefulmente.

**Confidence:** 0.78

**Evidencia:**
- Línea 700: `_log.warning(...)` — `_log` no está definido en el scope de `execute_tool_call`; el logger del módulo es `logger` en línea 36.
- `read_file` está envuelto en `asyncio.wait_for(timeout=15.0)` — stalls del filesystem Docker overlay pueden disparar `TimeoutError`.
- La respuesta "La respuesta tomó demasiado tiempo" coincide con brain.py línea ~1248 — terminación por timeout del loop de herramientas.
- Tiempos de respuesta muestran 25-31s en preguntas sobre pipelines — consistente con timeout de 15s + reintentos.
- La respuesta fabricada "File not found" — el bot en mal estado después de la propagación del NameError.

### Causas secundarias

1. **Sección ambigua de FILESYSTEM ACCESS**: `build_system_prompt()` deía "You can read the filesystem directly" — lenguaje ambiguo que hacía creer al modelo que tenía acceso directo al filesystem en lugar de acceso mediado por herramientas.

2. **Falta de regla de "admitir cuando no se puede"**: El system prompt carecía de una instrucción explícita: "If read_file returns an error or you cannot access a file, say 'I cannot access that file' instead of guessing" — lo que prevenía la fabricación cuando la herramienta fallaba.

3. **Modelo identidad confundida**: El modelo Claim "Gemini 3 Pro" — una sesión previa (20260408_010535) agregó la sección MODEL IDENTITY pero puede no haberse propagado correctamente al contenedor en ejecución.

## 4. Qué se Corrigió

### Fix 1 (CRÍTICO): `_log` → `logger` en TimeoutError handler
**Archivo:** `agent/brain.py`, línea ~700
**Cambio:** `_log.warning(...)` → `logger.warning(...)`
**Impacto:** Elimina el `NameError` que ocurría cada vez que `read_file` disparaba timeout. El logger a nivel de módulo es `logger` (línea 36). Este era un bug latente — si el filesystem de Docker se quedaba colgado lo suficiente para disparar el timeout de 15s, el bot cascadaía con `NameError` en lugar de manejar el error.

### Fix 2: Timeout de read_file 15s → 20s
**Archivo:** `agent/brain.py`, línea 144
**Cambio:** `_READ_FILE_TIMEOUT = 15.0` → `_READ_FILE_TIMEOUT = 20.0`
**Impacto:** Da más margen al filesystem de Docker overlay. 15s era demasiado agresivo para stalls del overlay. 20s permanece por debajo del timeout de ejecución de herramientas de MiniMax (~20-30s).

### Fix 3: Bloque BEHAVIORAL RULES en build_system_prompt()
**Archivo:** `agent/brain.py`
**Cambio:** Nueva sección `## BEHAVIORAL RULES (HIGH PRIORITY — ALWAYS FOLLOW)` después de la sección MODEL IDENTITY, conteniendo 3 reglas NEVER explícitas:
1. "NEVER claim to be Gemini, Claude, GPT, or any other AI model other than MiniMax-M2.7"
2. "NEVER invent n8n workflow names, session IDs, fix outcomes, or pipeline state from memory — always use n8n_list_workflows or read_file to verify"
3. "If asked about pipelines, workflows, session state, or any system query: ALWAYS use the available tools (n8n_list_workflows, read_file) BEFORE answering"
**Impacto:** Las reglas están posicionadas como constraints de alta prioridad cerca de la cima del prompt, no como contenido informativo enterrado. El modelo ahora tiene instrucciones hard-coded de usar herramientas ANTES de responder sobre queries del sistema.

### Fix 4: `_validate_tool_use_priority()` startup check
**Archivo:** `agent/brain.py`
**Cambio:** Nueva función de validación al inicio que parsea `build_system_prompt()` y verifica que contenga los 3 tokens requeridos: "MiniMax-M2.7", "NEVER invent", "n8n_list_workflows". Si falta alguno, levanta `AssertionError` con log `CRITICAL`.
**Impacto:** Si alguien edita `build_system_prompt()` y accidentalmente remueve las reglas, el contenedor falla al iniciar con un error claro.

### Fix 5: `_validate_read_file_timeout_value()` startup check
**Archivo:** `agent/brain.py`
**Cambio:** Nueva función que verifica `_READ_FILE_TIMEOUT >= 20.0` al inicio del módulo. Si el valor es menor, levanta `AssertionError`.
**Impacto:** Previene regresiones donde alguien baja el timeout a un valor inseguro.

## 5. Mejoras Estructurales Añadidas

1. **Validación de startup `_validate_tool_use_priority()`**: Assertion que falla al inicio del contenedor si las 3 reglas NEVER no están presentes en el system prompt. Asegura que la restricción sobreviva a reconstrucciones del contenedor.

2. **Validación de startup `_validate_read_file_timeout_value()`**: Assertion al inicio que verifica el timeout de read_file no esté por debajo de 20s.

3. **Ampliación de suite de tests**: Se agregaron 3 nuevas clases de tests a `tests/test_pipeline_architecture.py`:
   - `TestBehavioralRulesPriority` (4 tests)
   - `TestReadFileTimeoutValue` (2 tests)
   - `TestOrchestratorPathAccessible` (1 test)
   Total: **15/15 tests PASS**.

4. **Logs de observabilidad**:
   - `TOOL_USE_PRIORITY_VALIDATION PASSED` o `CRITICAL: TOOL_USE_PRIORITY_VALIDATION FAILED`
   - `READ_FILE_TIMEOUT_VALIDATION PASSED: timeout=20.0s`
   - `_log.warning` reemplazado por `logger.warning` (ahora funciona correctamente en logs)

5. **Documentación actualizada**: CHANGELOG.md, agent_memory.md, y pipeline/CLAUDE.md todos actualizados con la sesión 20260408_055435.

## Resumen Ejecutivo

El bot estaba fabricando respuestas sobre el sistema porque cuando la herramienta `read_file` se agotaba por timeout (filesystem Docker overlay), el bloque de error en `execute_tool_call()` lanzaba un `NameError` adicional (`_log` no estaba definido). Este segundo error se propagaba sin manejo, dejando al bot sin respuesta válida — y en lugar de admitir que no podía acceder a los archivos, el modelo (sin reglas explícitas de behavioral constraints) entrenaba conocimiento y respondía con fabricate. La corrección fue: (1) fix del `_log`→`logger` (1 línea), (2) reglas NEVER explícitas en el prompt como constraints de alta prioridad, (3) timeout aumentado a 20s, (4) validaciones de startup para ambas correcciones. **Container healthy, 15/15 tests passing, 0 errores nuevos.**
