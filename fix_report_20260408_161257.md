## 3. Causa Raíz

**Causa técnica principal:** La sección `PIPELINE ARCHITECTURE` en `build_system_prompt()` (`brain.py`, líneas 292-299) solo decía que el pipeline es "a Python CLI automation system" sin especificar la estructura de 7 agentes, nombres de agentes, roles, convención de nombres de archivos de sesión, ni reglas de comportamiento para consultas de estado del pipeline. MiniMax LLM treat this como información suave (soft context) y recurrió a sus datos de entrenamiento interno, produciendo respuestas confidently incorrectas — rutas de archivos inventadas, afirmación de que no existía herramienta de shell.

**Causa secundaria:** La función `classify_intent()` (`brain.py`, líneas 55-67) solo reconocía keywords explícitos de fix ("corrígelo", "mejora", "fix"). Las consultas sobre estado del pipeline ("revisa el archivo 07", "por qué no llegas al Agente 7", "estado del pipeline") eran clasificadas como 'chat' — negándole al bot acceso al contexto de `execute_self_fix` que le permitiría investigar el estado del pipeline.

**Tercera causa:** `execute_self_fix()` (`self_improve.py`, línea ~595) retornaba el mensaje "Te notificaré cuando esté lista (~2-5 min)" — una promesa de notificación futura que nunca se cumple. No existe código que envíe un mensaje de Telegram cuando el pipeline termina.

**Detalles técnicos exactos:**
- Archivo: `/root/minimax-agent/agent/brain.py`
- Función: `build_system_prompt()` — sección PIPELINE ARCHITECTURE
- Líneas: 292-299 (antes del fix)
- Mecanismo: La sección genérica no proveía tokens que anclaran el conocimiento; el LLM free-formed basándose en entrenamiento

**Confianza:** 0.87

## 4. Qué se Corrigió

**Paso 1 — Expandir sección PIPELINE ARCHITECTURE en `build_system_prompt()`:**
Se reemplazó la descripción genérica con una tabla completa de 7 agentes:
- Agent 1 (AGENT_1): Failure Analyst → 01_failure_trajectory.md
- Agent 2 (AGENT_2): Root Cause Finder → 02_root_cause.md
- Agent 3 (AGENT_3): Implementation Planner → 03_implementation_plan.md
- Agent 4 (AGENT_4): Executor → 04_execution_report.md
- Agent 5 (AGENT_5): Results Auditor → 05_audit_verdict.md
- Agent 6 (AGENT_6): Decision Maker → 06_decision.md
- Agent 7 (AGENT_7): Debug Approver + GitHub Publisher → 07_debug_report.md

También se incluyó la convención de nombres de archivos de sesión (`00_trigger.md` a `07_debug_report.md`) y reglas de comportamiento que instruyen a MiniMax a usar `read_file` para consultar estado del pipeline en lugar de fabricar respuestas.

**Paso 2 — Agregar keywords específicos de pipeline a `classify_intent()`:**
Se añadieron 6 keywords a `IMPROVEMENT_KEYWORDS`: "agente 7", "sesión del pipeline", "archivo 07", "reporte del pipeline", "estado del pipeline", "pipeline status", "agente siete". Ahora queries como "por qué no llegas al Agente 7" se clasifican como 'improve' y acceden al contexto de `execute_self_fix` con la herramienta `read_file` disponible.

**Paso 3 — Agregar validación `_validate_pipeline_agent_roles()`:**
Nueva función en `brain.py` que verifica al inicio que el system prompt contenga todos los tokens AGENT_1 a AGENT_7. Si falta alguno, el contenedor falla inmediatamente con `AssertionError` y log `CRITICAL: PIPELINE_AGENT_ROLES_VALIDATION FAILED`.

**Paso 4 — Agregar validación `_validate_classify_intent_pipeline_keywords()`:**
Nueva función que verifica que `IMPROVEMENT_KEYWORDS` contenga los keywords de pipeline requeridos. Falla con `CRITICAL: CLASSIFY_INTENT_PIPELINE_KEYWORDS_VALIDATION FAILED` si falta alguno.

**Nota sobre incidente de rebuild:** El primer rebuild de Docker falló con `NameError: name 'YYYYMMDD_HHMMSS' is not defined` porque `{YYYYMMDD_HHMMSS}` fue interpretado como expresión Python dentro de un f-string. Se corrigió usando double-brace escaping: `{{YYYYMMDD_HHMMSS}}`. El segundo rebuild fue exitoso.
