## 3. Causa Raíz

### Causa técnica

**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Función:** `_validate_odoo_bridge_endpoints()` (líneas 234-271)

**Mecanismo:** La función de validación de startup enviaba `json={}` (diccionario vacío) a TODOS los endpoints POST del microservicio Odoo como "probe" de conectividad. Los endpoints que tienen campos Pydantic requeridos (como `name`, `partner_id`, `items`, etc.) rechazan payloads vacíos con HTTP 422 (Unprocessable Entity). El código capturaba `except httpx.HTTPError` — que incluye 422 — y contaba todos los errores HTTP como fallos de infraestructura. Como 10 de 12 endpoints tienen campos requeridos, el 83% fallaba con 422, disparando la alerta CRITICAL.

El microservicio estaba completamente sano (21 horas uptime, respondiendo correctamente). Los errores 422 eran el comportamiento esperado de Pydantic rechazando payloads vacíos.

### Problemas secundarios encontrados

1. **`agent_chat()` ruta incorrecta:** Llamaba `POST /agent/chat` pero el microservicio tiene `POST /agent_chat` — cualquier llamada real retornaría 404.

2. **`batch_price()` esquema incorrecto:** Enviaba `{"search_terms": lista}` pero el modelo `BatchPriceRequest` del microservicio espera `{"items": [...]}` — las llamadas reales también fallarían con 422.

3. **`update_opportunity` retorna 500:** El microservicio devuelve 500 Internal Server Error — esto es un bug pre-existente del handler o del backend de Odoo, no causado por este fix.

4. **`/agent_chat` no existe en el microservicio:** El endpoint nunca fue implementado en `main.py` del microservicio (verificado en líneas 1-2329).

### Explicación accesible para Fernando

El sistema tiene un "check de salud" que verifica si el servicio de Odoo está funcionando. Cuando el bot se inicia, hace pequeñas peticiones de prueba a cada endpoint del servicio interno. El problema era que enviaba peticiones vacías (`{}`) a endpoints que requieren datos obligatorios (como el nombre del contacto). El servicio correctamente rechazaba estas peticiones vacías diciendo "422 — falta información requerida", pero el check interpretaba TODOS los rechazos como "el servicio está caído", disparando 13 alertas falsas de CRITICAL en 2 horas.

## 4. Qué se Corrigió

### Cambio 1: `_validate_odoo_bridge_endpoints()` — payloads por endpoint + clasificación de 422

**Antes:**
```python
r = await c.post(url, headers=_headers(), json={})  # empty dict to ALL endpoints
# ... all httpx.HTTPError counted as failure
```

**Después:**
- Se creó `VALID_PAYLOAD_ENDPOINTS` con payloads mínimos apropiados por endpoint
- Se separaron los contadores: `failed` (errores de conexión/503) vs `validation_err` (422)
- HTTP 422 ahora se clasifica como `VALIDATION_ERROR` (log INFO, no se cuenta como fallo)
- Solo 400/403/404/503 se cuentan como errores de infraestructura
- Log final: `"startup validation passed — {failed}/{total} infrastructure errors, {validation_err}/{total} schema-validated (OK)"`

### Cambio 2: `agent_chat()` — corregir ruta `/agent/chat` → `/agent_chat`

**Línea 211:**
```python
# Antes:
f"{ODOO_URL}/agent/chat"
# Después:
f"{ODOO_URL}/agent_chat"
```

### Cambio 3: `batch_price()` — corregir clave JSON `search_terms` → `items`

**Línea 184:**
```python
# Antes:
json={"search_terms": search_terms}
# Después:
json={"items": search_terms}
```

## Resumen Ejecutivo

**Problema:** 13 alertas CRITICAL en 2 horas indicando que el 83% de los endpoints del microservicio Odoo estaban caídos — falso positivo.

**Causa raíz:** La función `_validate_odoo_bridge_endpoints()` enviaba payloads vacíos `{}` a todos los endpoints POST. Los endpoints con campos requeridos retornaban 422 (rechazo de schema por Pydantic), y el código contaba TODOS los errores HTTP como fallos de infraestructura.

**Fix aplicado:** (1) Payloads específicos por endpoint, (2) HTTP 422 clasificado como VALIDATION_ERROR (no fallo), (3) ruta `agent_chat` corregida `/agent/chat` → `/agent_chat`, (4) clave `batch_price` corregida `search_terms` → `items`.

**Resultado:** Los CRITICAL falsos positivos están eliminados — logs muestran `VALIDATION_ERROR` para 422s y mensaje "passed" final. Contenedor Docker reconstruido y healthy.

**Issues pendientes (no causados por este fix):** `agent_chat` retorna 404 porque el endpoint no existe en el microservicio; `update_opportunity` retorna 500 (bug del microservicio). Ambos requieren cambios en el microservicio FastAPI, no en el bridge.
