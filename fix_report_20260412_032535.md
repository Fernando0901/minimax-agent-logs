# Fix Report — partner_name vs partner_id field contract mismatch in create_opportunity

---

## 1. Problema Reportado

El monitor proactivo de salud detectó **3 errores recurrentes** del tipo `VALIDATION_ERROR (422)` en el endpoint `/create_opportunity` del microservicio Odoo, con primera ocurrencia en `2026-04-12 02:55:43,929`. El error se repetía en el arranque del minimax-agent durante la validación de salud de `odoo_bridge`, específicamente en la sonda de `create_opportunity`. La traza indicaba:

```
INFO [odoo_bridge] startup check create_opportunity: VALIDATION_ERROR (422)
```

El endpoint estaba devolviendo **HTTP 422 Unprocessable Entity** — lo que significa que FastAPI/Pydantic rechazaba el payload ANTES de que el request llegara al handler del endpoint.

## 3. Causa Raíz

### Versión Accessible (Español)

**Problema:** El cliente HTTP `odoo_bridge.py` enviaba un campo llamado `partner_name` (nombre del socio como texto), pero el esquema Pydantic `CreateOppRequest` en el microservicio FastAPI esperaba un campo llamado `partner_id` (identificador numérico o de texto, OBLIGATORIO sin valor por defecto). Cuando el JSON `{"partner_name": "test", ...}` llegaba a FastAPI, Pydantic no encontraba la clave `partner_id` obligatoria y rechazaba el request con HTTP 422 **antes** de que el cuerpo de la petición llegara al handler del endpoint.

**Detalle técnico:**
- `odoo_bridge.py:100` — el JSON enviado era `{"partner_name": partner_name, ...}`
- `main.py:820` — el esquema define `partner_id: Union[int, str]` (requerido, sin default, sin campo `partner_name`)
- El código del handler en la línea 1072 (`resolve_id_strict(models, uid, "res.partner", request.partner_id, "Customer")`) **nunca se ejecutaba** porque Pydantic rechazaba el request antes
- La misma discrepancia existía en `create_quotation()` (línea 154 de odoo_bridge.py) y en ambas sondas de validación de arranque (líneas 248-249)
- El microservicio nunca fue modificado para recibir `partner_name` — el esquema se actualizó a `partner_id` pero `odoo_bridge.py` nunca se sincronizó

**Confianza del diagnóstico:** 0.95

## 4. Qué se Corrigió

### Cambios en el Microservicio (`/opt/Agente_Odoo_V2/Deployment_Package/Microservice/main.py`)

1. **CreateOppRequest schema** — Se agregó `partner_name: Optional[str] = None` como campo opcional adicional a `partner_id`. Ahora acepta ambos.
2. **Handler de create_opportunity** — Ahora verifica si `partner_id` es None y `partner_name` fue proporcionado. Si es así, resuelve `partner_name` a `partner_id` usando `resolve_id_strict()` internamente antes de llamar a Odoo. Incluye logging INFO cuando usa el camino alternativo.
3. **CreateQuotationRequest schema** — Mismo patrón: `partner_id: Optional[int] = None` + `partner_name: Optional[str] = None`.
4. **Handler de create_quotation** — Mismo patrón de resolución con logging.

### Cambios en el Cliente (`/root/minimax-agent/agent/odoo_bridge.py`)

5. **create_opportunity()** — El parámetro de función se renombró de `partner_name` a `partner_id`. El JSON enviado cambió de `"partner_name": partner_name` a `"partner_id": partner_id`.
6. **create_quotation()** — El JSON enviado cambió de `"partner_name": partner_name` a `"partner_id": partner_id` (el parámetro de función se mantuvo como `partner_name` por compatibilidad backwards).
7. **Sondas de validación de arranque** — Ambas sondas ahora envían `partner_id: "test"` en lugar de `partner_name: "test"`.

### Nuevas Pruebas

8. **Test suite creada:** `/root/minimax-agent/tests/test_odoo_bridge_field_contract.py` — 9 tests covering:
   - La firma de `create_opportunity()` usa `partner_id`
   - El JSON payload de `create_opportunity()` envía `partner_id`
   - El JSON payload de `create_quotation()` envía `partner_id`
   - Las sondas de arranque usan `partner_id` en ambos endpoints
   - Los schemas de request aceptan ambos campos (`partner_id` y `partner_name`)

**Resultado:** 9/9 tests PASS.

### Contenedores

- **minimax-agent:** Reconstruido y restarted — Up (healthy)
- **odoo_python_agent:** Reconstruido y restarted — Up (healthy, 4 segundos al momento del reporte)

## 5. Mejoras Estructurales Añadidas

### Prevención de Recurrencia

- Si el esquema del microservicio alguna vez elimina el soporte para `partner_name`, la sonda de arranque fallará con un error diferente (missing required field) — no HTTP 422 de validación de esquema — haciendo la regresión inmediatamente visible.
- La lógica de resolución interna en el handler ahora hace que ambos campos (`partner_id` o `partner_name`) seanfirst-class citizens, eliminando la posibilidad de que un mismatch de nombre de campo cause rechazos silenciosos.

### Capa de Validación

- La clasificación de HTTP 422 como `VALIDATION_ERROR` (no failure) en `_validate_odoo_bridge_endpoints()` ya maneja graciosamente estos errores — no disparan CRITICAL ni afectan el estado de salud del agente.

### Observabilidad

- Ambos handlers ahora emiten logs INFO cuando usan el camino alternativo `partner_name`:
  `"[create_opportunity] partner_name provided, resolving to partner_id"`
- Ambos打印 el `partner_id` resuelto antes de llamar a Odoo.
