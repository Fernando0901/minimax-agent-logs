## 3. Causa Raíz

### Causa raíz técnica

**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Funciones afectadas:** `schedule_activity()` (líneas 133-149), `create_quotation()` (líneas 152-168)
**Mecanismo:** El cliente HTTP `odoo_bridge.py` envía cargas útiles JSON con **nombres de campo que no coinciden con los nombres de campo de los modelos Pydantic** del microservicio FastAPI.

**Errores específicos:**

1. **`schedule_activity()`** enviaba `model`/`record_id`/`activity_type` pero el microservicio `ScheduleActivityRequest` (main.py:878-906) espera `res_model`/`res_id`/`activity_type_id`. Esta era una **doble discordancia de nombres de campo**.

2. **`create_quotation()`** enviaba `order_lines` pero el microservicio `CreateQuotationRequest` (main.py:1864) espera `lines`. Además, la función aceptaba el parámetro `partner_name` pero **nunca lo incluía en la carga útil JSON** — cuando `partner_id` era una cadena vacía `""`, Python evaluaba `"" is not None` como `True`, pasando la cadena vacía a Odoo causando error de dominio → HTTP 500.

3. **Sondas de validación de startup** para `schedule_activity` usaban nombres de campo incorrectos causando 422s en las verificaciones de salud al iniciar.

4. **Respuesta lenta extrema** (33,387ms): el pool de conexiones Odoo (`_odoo_pool`, máximo 5 conexiones, TTL 300s) combinado con `asyncio.to_thread` para llamadas Odoo XML-RPC bloqueantes causaba acumulación en cola bajo carga concurrente.

### Contexto histórico
- Este era el error **número 19 de recurrencia** del mismo tipo (`odoo_bridge field name mismatch`).
- La sesión `20260412_032535` había corregido parcialmente: añadió `partner_name: Optional[str] = None` a los esquemas `CreateOppRequest`/`CreateQuotationRequest`, pero **no corrigió** las discordancias de nombres de campo en `schedule_activity`, ni que `create_quotation` ignoraba `partner_name` en la carga útil, ni `order_lines` vs `lines`.
- Tasa histórica de éxito era **0.0%** para este patrón antes de esta sesión.

## 4. Qué se Corrigió

### Cambio 1: `schedule_activity()` — nombres de campo corregidos
**Antes (líneas 145-149):**
```python
json={"record_id": record_id, "model": model,
      "activity_type": activity_type, "summary": summary,
      "date_deadline": date_deadline}
```
**Después:**
```python
json={"res_id": res_id, "res_model": res_model,
      "activity_type_id": activity_type_id, "summary": summary,
      "date_deadline": date_deadline}
```

### Cambio 2: `create_quotation()` — partner_name ahora se envía, lines no order_lines
**Antes:**
```python
payload = {"partner_id": partner_id, "order_lines": lines}
```
**Después:**
```python
payload = {"lines": lines}
if partner_id:
    payload["partner_id"] = partner_id
else:
    if partner_name:
        payload["partner_name"] = partner_name
        logger.info(f"[odoo_bridge] create_quotation: using partner_name='{partner_name}'")
    else:
        payload["partner_id"] = partner_id
```

### Cambio 3: Sondas de validación de startup corregidas
Las sondas en `_validate_odoo_bridge_endpoints()` ahora envían nombres de campo correctos para `schedule_activity` y `create_quotation`.

### Cambio 4: Suite de tests creada
Nuevo archivo `/root/minimax-agent/tests/test_odoo_bridge_field_names.py` con 6 tests que validan la corrección de nombres de campo.

### Contenedor Docker rebuilt
`minimax-agent` reconstruido y verificado como **healthy** (56 segundos post-rebuild).
