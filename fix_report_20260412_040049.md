# Fix Report — Sondas de validación de startup de odoo_bridge enviaban payloads incompatibles con los esquemas Pydantic

## 1. Problema Reportado

El monitor proactivo de salud detectó 6 errores recurrentes de tipo `VALIDATION_ERROR (422)` en las últimas 2 horas, todos originados en `[odoo_bridge] startup check search`. Los logs mostraban:

```
2026-04-12 03:55:39,213 INFO [odoo_bridge] startup check search: VALIDATION_ERROR (422)
2026-04-12 03:55:42,519 INFO [odoo_bridge] startup check create_quotation: VALIDATION_ERROR (422)
```

El sistema reportaba un 83% de endpoints caídos (10 de 12), lo que activaba el umbral CRITICAL del 50%. Sin embargo, el problema real no era que el microservicio FastAPI estuviera caído — era que las **sondas de validación de startup enviaban payloads estructuralmente incorrectos** que causaban rechazos Pydantic (HTTP 422).

Este era un problema recurrente con **19 repeticiones y 0% de tasa de éxito en soluciones previas** — las sesiones anteriores (20260412_034628, 20260412_032535, 20260412_024519) habían corregido el código de los clientes reales (`search()`, `create_quotation()`) pero **NO las sondas de startup** en `_validate_odoo_bridge_endpoints()`. Esto explicaba por qué los errores 422 persistían a pesar de los "fixes" aplicados.

## 3. Causa Raíz

### Detalle Técnico (para el registro del pipeline)

**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Función:** `_validate_odoo_bridge_endpoints()` (líneas 285-305 para la lista `all_endpoints`, líneas 311-332 para el loop de ejecución)

**Mecanismo identificado:**

La función `_validate_odoo_bridge_endpoints()` envía payloads de prueba (sondas) a cada endpoint del microservicio FastAPI para validar que la conexión está sana. El clasificador de errores trata **TODOS** los HTTP 422 como "VALIDATION_ERROR esperados" y los **excluye del conteo de fallos hacia el umbral CRITICAL** (50%). Esto significaba que incluso cuando las sondas enviaban payloads radicalmente incorrectos — generando 422 por campos requeridos faltantes o tipos wrong — el sistema reportaba "validación de startup PASSED" porque los 422 no contaban.

Las 5 sondas defectuosas:

1. **`search`** (línea 286): Enviaba `{}` pero `SearchRequest.search_term: str` es **requerido sin default y sin Optional wrapper** — Pydantic rechaza cuerpo vacío → HTTP 422 ✅ Ahora envía `{"search_term": "test", "limit": 5}`

2. **`get_product_price`** (línea 287): Enviaba `{}` pero el handler requiere `search_term` no vacío → HTTP 400 (lógica de negocio, counted correctly)

3. **`get_product_bom`** (línea 288): Mismo patrón que `get_product_price` → HTTP 400

4. **`create_quotation`** (línea 298): Enviaba `{"partner_id": "test", "lines": []}` pero `CreateQuotationRequest.partner_id: Optional[int]` — string `"test"` falla la coerción int de Pydantic → HTTP 422 ✅ Ahora envía `{"partner_name": "test", "lines": []}`

5. **`overdue_activities`** (línea 299): Enviaba `{}` pero el handler requiere `user_id: int` → HTTP 400 (lógica de negocio) ✅ Ahora envía `{"user_id": 1, "limit": 5}`

**Raíz del bug:** El fix anterior (sesión 20260412_034628) corrigió los payloads en las **funciones cliente reales** (`search()`, `create_quotation()`) pero NO las **sondas de startup** en `_validate_odoo_bridge_endpoints()`. Las sondas nunca fueron actualizadas, así que seguían generando 422s que eran classification como "esperados" y excluidos del umbral CRITICAL — haciendo que el sistema pareciera sano cuando en realidad las sondas estaban defectuosas.

**Archivo del microservicio:** `odoo_python_agent` container, `/app/main.py`
- `SearchRequest` (línea 696): `search_term: str` — requerido sin default
- `CreateQuotationRequest` (línea 1858): `partner_id: Optional[int]`
