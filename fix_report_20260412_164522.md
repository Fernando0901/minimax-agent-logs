# Fix Report — Odoo connection pool exhaustion causing ReadTimeout errors

## 1. Problema Reportado

El Monitor de Salud Proactivo detectó 7 errores recurrentes de tipo `CONNECTION_ERROR — ReadTimeout` en el módulo `odoo_bridge` del MiniMax-Agent durante las últimas 2 horas, específicamente a las 16:40:14 del 12 de abril de 2026. Los errores afectaron múltiples endpoints del microservicio Odoo (`search`, `create_opportunity`, `update_opportunity`, `schedule_activity`), y la tendencia de tiempos de respuesta mostraba picos de hasta **169 segundos** a las 15:44:13 — muy por encima del timeout de 30 segundos configurado en el cliente httpx.

Este era un **nuevo modo de fallo** dentro de la familia de errores INTEGRATION_ERROR. Las 23 ocurrencias históricas previas habían sido corregidas mediante ajustes de nombres de campos y payloads de validación en los probes de startup. Esta era la primera vez que el problema era el agotamiento real del pool de conexiones de Odoo, no un error de esquema del lado del cliente.

## 3. Causa Raíz

### Descripción Accesible (español)

El sistema tiene un **pool de conexiones a Odoo** con capacidad para 5 conexiones simultáneas. Cada conexión permanece en el pool durante 300 segundos (5 minutos) una vez creada. Cuando Odoo se vuelve lento o se cuelga en sus llamadas XML-RPC, las conexiones del pool se ocupan durante períodos muy largos (30-169 segundos observados). Con solo 5 conexiones disponibles y múltiples solicitudes concurrentes, el pool se **agota rápidamente**. Las nuevas solicitudes se quedaban esperando indefinidamente detrás del semáforo del pool, o fallaban inmediatamente si el pool estaba completamente saturado.

Además, el cliente HTTP (`odoo_bridge.py`) tenía un timeout plano de 30 segundos para todas las operaciones — pero Odoo estaba tardando hasta 169 segundos en responder. El timeout de 30 segundos era **demasiado agresivo**, causando errores `ReadTimeout` prematuramente mientras las conexiones del pool seguían ocupadas. Y no había lógica de reintento: cualquier error transitorio se propagaba directamente al usuario.

### Detalle Técnico

**Archivo:** `/opt/Agente_Odoo_V2/Deployment_Package/Microservice/main.py`
**Función:** `OdooConnectionPool.get_connection()` (líneas 246-286)

```
Mecanismo: OdooConnectionPool(max_connections=5, connection_ttl=300)
  → threading.Semaphore(5) sin timeout en acquire()
  → Cuando Odoo XML-RPC se cuelga (169s observadas):
      1. get_connection() bloquea en semaphore.acquire() indefinidamente
      2. O bien las solicitudes se encolan hasta que el pool libera conexiones a los 300s TTL
      3. O bien el cliente httpx agota su timeout de 30s y mata la lectura
         PERO la conexión del pool NO se libera — permanece ocupada hasta TTL
  → PoolExhaustedError o ReadTimeout propagados al usuario
```

**Archivo secundario:** `/root/minimax-agent/agent/odoo_bridge.py`
- `httpx.AsyncClient(timeout=30)` — timeout plano de 30s para TODAS las funciones
- Sin reintento en errores transitorios (`ReadTimeout`, `ConnectError`)
- `/search` (el endpoint más pesado, 3 llamadas Odoo secuenciales) recibía el mismo timeout de 30s que operaciones simples

## 4. Qué se Corrigió

### A nivel del microservicio Odoo (`main.py`):

1. **`PoolExhaustedError`** — Nueva clase de excepción que se lanza cuando el pool no puede entregar una conexión dentro del tiempo máximo de espera.

2. **`max_wait=30`** — Parámetro agregado a `OdooConnectionPool.__init__()`. El `semaphore.acquire(timeout=self._max_wait)` ahora espera máximo 30 segundos. Si no hay conexión disponible, levanta `PoolExhaustedError` en lugar de bloquearse indefinidamente.

3. **Slow-call tracking** — Diccionario `_slow_calls` que registra conexiones que tomaron más de 30 segundos en completarse. La próxima vez que se pida una conexión, las conexiones marcadas como lentas se skippean y se fuerza creación de una conexión nueva.

4. **Pool exhaustion WARNING** — Cuando `pool_size >= max_connections`, se emite un log WARNING con las estadísticas del pool para facilitar monitoreo.

5. **`_validate_connection_pool_max_wait()`** — Check de startup que verifica que `max_wait=30`, `_slow_calls` dict existe, `_slow_call_threshold=30`, y `get_stats()` devuelve los campos requeridos. El contenedor no inicia si esta validación falla.

### A nivel del bridge (`odoo_bridge.py`):

6. **Timeouts aumentados** — Todas las 12 funciones ahora usan `httpx.Timeout(120.0, connect=10.0, read=120.0)` (2 minutos para operaciones normales). La función `search()` usa `httpx.Timeout(180.0, connect=10.0, read=180.0)` (3 minutos) para reflejar las operaciones pesadas observadas de hasta 169 segundos.

7. **`_retry_with_backoff()`** — Helper que captura `httpx.ReadTimeout` y `httpx.ConnectError`, reintenta hasta 2 veces con delays exponenciales (2s, 4s), y retorna error dict después de agotar los reintentos. Las 12 funciones fueron refactorizadas para usar este wrapper.

8. **`_validate_odoo_bridge_timeout_values()`** — Check de startup basado en AST que verifica que ninguna función usa timeout plano < 60s, y que `_SEARCH_TIMEOUT` existe con `read=180`. Se ejecuta al importar el módulo y al hacer startup.

## 5. Mejoras Estructurales Añadidas

| Mejora | Descripción |
|--------|-------------|
| **Circuit Breaker Pattern** | `PoolExhaustedError` + `max_wait=30s` implementa un circuit breaker real: cuando el backend de Odoo se degrada, las solicitudes fallan rápido (después de 30s de espera) en lugar de colgarse indefinidamente. |
| **Slow-Call Detection** | Las conexiones que toman >30s se marcan como potencialmente "stale". El sistema las skippea en la siguiente adquisición y fuerza creación de conexión limpia — previene que conexiones realmente colgadas contaminen el pool. |
| **Retry con Exponential Backoff** | 2 reintentos con delays de 2s y 4s para errores transitorios. Los errores de Odoo son frecuentemente transitorios (timeouts de red, carga temporal). Este retry previene que un spike aislado colapse la experiencia del usuario. |
| **Validation Layer Dual** | `_validate_connection_pool_max_wait()` en el microservicio Y `_validate_odoo_bridge_timeout_values()` en el bridge. Ambos fallan el arranque del contenedor si las validaciones no pasan — previene deploys con configuraciones incorrectas. |
| **Observabilidad** | Pool exhaustion → WARNING con stats; slow call detectada → WARNING; reintentos → INFO/WARNING; ambos contenedores emiten logs de startup confirmando que las validaciones pasaron. |

## 7. Issues Pre-existentes Encontrados

Estos issues fueron identificados durante la auditoría de Agent 5 y **no fueron introducidos por este fix**. Están documentados para seguimiento futuro:

### HIGH — Errores HTTP 500 del Backend de Odoo
**Descripción:** `create_opportunity`, `update_opportunity`, y `schedule_activity` están devolviendo HTTP 500 con errores del backend de Odoo:
- `create_opportunity` → `psycopg2.errors.InvalidTextRepresentation` (problema de esquema de datos)
- `update_opportunity` → `Record does not exist or has been deleted` (problema de datos)
- `schedule_activity` → `No se pudo completar... restricciones de seguridad` (problema de ACL)

**Acción recomendada:** Investigar el esquema XML-RPC de Odoo y los permisos ACL — es una familia de errores separada del agotamiento del pool.

### MEDIUM — Desajuste de Ruta `/agent/chat` vs `/agent_chat`
**Descripción:** `minimax-agent` llama a `/agent_chat` (sin slash inicial) causando HTTP 404, mientras el endpoint está registrado como `/agent/chat` (con slash).

**Acción recomendada:** Corregir la ruta del cliente en `odoo_bridge.py` o verificar el registro del endpoint en `odoo-python-agent`.
