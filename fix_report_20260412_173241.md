### Agente 2 — Análisis de Causa Raíz
Lectura completa del código fuente de `/root/minimax-agent/agent/odoo_bridge.py` (501 líneas) y del microservicio FastAPI en `/opt/Agente_Odoo_V2/Deployment_Package/Microservice/main.py`.

**Hallazgos técnicos:**

1. **Ruta incorrecta `/agent_chat` vs `/agent/chat` (líneas 336 y 459):**
   La función cliente `agent_chat()` en odoo_bridge.py llama a `POST /agent_chat`. El microservicio Odoo tiene registrado `POST /agent/chat` (main.py:2766). Esta ruta incorrecta siempre devuelve HTTP 404.

2. **`partner_id: "test"` (string) en payload de `create_opportunity` (línea 430):**
   El payload de validación de startup es `{"name": "test", "partner_id": "test", ...}`. El modelo `crm.lead` de Odoo requiere un `partner_id` de tipo integer. Odoo rechaza el string `"test"` con HTTP 500 (`psycopg2.errors.InvalidTextRepresentation`).

3. **`partner_id: "test"` (string) en payload de `create_quotation` (línea 436):**
   Mismo problema — el payload envía `"test"` como partner_id, causando HTTP 500.

4. **Cascada hacia la salud del contenedor:**
   Cuando las operaciones de Odoo son lentas (169s observado), el event loop de asyncio se bloquea. El healthcheck de Docker es un dummy (`python -c "import sys; sys.exit(0)"`) que siempre retorna 0 — no detecta el starve del event loop.

## 3. Causa Raíz

### Causa Raíz Técnica (en español, accesible para Fernando)

El módulo `odoo_bridge.py` tiene dos tipos de errores en los payloads de validación de startup:

**Error 1 — Tipo de dato incorrecto:**
Los probes de startup para `create_opportunity` y `create_quotation` enviaban `partner_id: "test"` donde Odoo espera un entero. Esto es como enviar texto donde Odoo espera un número de identificación de cliente. Odoo responde con error 500 porque no puede convertir "test" a número.

**Error 2 — Ruta de API incorrecta:**
La función `agent_chat()` y el probe de startup llamaban a `/agent_chat` pero el microservicio tiene el endpoint en `/agent/chat` (con barra en medio). Siempre devuelve 404 porque la ruta no existe.

Estos errores causaban que 3 de 12 probes fallaran con errores de infraestructura (no de validación), elevando el rate de fallo a >50% y disparando el log CRITICAL. El contenedor no estaba realmente caído — los errores eran falsos positivos.

### Mecanismo Exacto
- **Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
- **Función:** `_validate_odoo_bridge_endpoints()` (líneas 419-491)
- **Dicts afectados:** `VALID_PAYLOAD_ENDPOINTS` (líneas 428-438) y `all_endpoints` (líneas 441-461)
- **Líneas específicas:** 336 (ruta en función cliente), 430 (partner_id en create_opportunity), 436 (partner_id en create_quotation), 459 (ruta en probe de startup)

## 5. Mejoras Estructurales Añadidas

### Capa de Validación — `_validate_probe_payloads()`
Nueva función de validation en startup que usa AST (Abstract Syntax Tree) para verificar que todos los payloads en `VALID_PAYLOAD_ENDPOINTS` tengan los tipos correctos:
- `partner_id` debe ser `int`, no `str`
- `search_term` debe ser `str`
- `user_id` debe ser `int`

Si un payload tiene tipos incorrectos, la función lanza `AssertionError` con mensaje descriptivo y hace log de `ODOO_BRIDGE_PROBE_PAYLOAD_VALIDATION FAILED`. Esto previene que el contenedor arranque con payloads incorrectos.

### Observabilidad
- Log `ODOO_BRIDGE_PROBE_PAYLOAD_VALIDATION PASSED` confirma que la validación pasó en cada restart
- Los errores HTTP 500 de Odoo ahora se distinguen de los errores de conexión
- Los errores HTTP 404 de ruta incorrecta ahora se identifican inmediatamente en los logs

### Prevención de Recurrencia
La combinación de `_validate_probe_payloads()` (verificación en startup) + la suite de tests (ejecutable manualmente) asegura que:
1. Un humano puede verificar que los payloads son correctos antes de commit
2. El contenedor no arranca si los payloads tienen tipos incorrectos
3. Un error de tipo en el futuro disparará el log CRITICAL inmediatamente, no después de horas
