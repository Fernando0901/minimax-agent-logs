# Fix Report — MiniMax context window exhaustion from conversation store bloat

## 1. Problema Reportado

El contenedor Docker `minimax-agent` reportó estado **unhealthy** (no saludable). El monitor de salud proactivo detectó que el campo de estado de salud estaba vacío/en blanco cuando fue reportado. El análisis de logs mostró que el agente de MiniMax respondía con tiempos excesivos (hasta 18 segundos) y eventualmente retornaba errores `base_resp.status_code: 2013 — context window exceeds limit` (límite de ventana de contexto excedido), indicando que la API de MiniMax estaba rechazando solicitudes porque el historial de conversación acumulado excedía su capacidad de contexto.

La alerta incluyó 11 ocurrencias en 2 horas del error:
```
2026-04-12 16:40:20,687 CRITICAL [odoo_bridge] STARTUP VALIDATION FAILED — 58% endpoints down (7/12)
```
Aunque este error de odoo_bridge era un problema secundario (fallos pre-existentes en el backend de Odoo), la causa raíz del fallo del contenedor era la incapacidad del agente de procesar mensajes debido al contexto de MiniMax agotado.

## 3. Causa Raíz

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Función:** `load_conversation()` (líneas 278-287)
**Mecanismo:** El historial de conversación se truncaba a `SLIDING_WINDOW=20` mensajes **solo al guardar** (línea 285), pero **no al cargar**. Esto significaba que si el archivo `conversation_store.json` crecía más allá de 20 mensajes por acumulación de ejecuciones previas o estado persistido, `load_conversation()` retornaba el historial completo sin truncar. Este historial completo se incluía en cada solicitud a la API de MiniMax vía el parámetro `messages` (línea 415-417), causando que eventualmente la API excediera su límite de ventana de contexto.

**Evidencia técnica:**
1. `load_conversation()` (líneas 278-287) **NO** aplicaba `SLIDING_WINDOW` al cargar — solo al guardar
2. `call_minimax()` enviaba el historial completo cargado en cada solicitud (línea 415-417)
3. Log de error: `"base_resp": {"status_code": 2013, "status_msg": "invalid params, context window exceeds limit"}}` — la API de MiniMax reportó explícitamente contexto excedido
4. Escalamiento de tiempos de respuesta de 5.8s → 18s durante 10 minutos — patrón de acumulación gradual
5. 25 recurrencias con 0% de éxito histórico sugiere que las correcciones previas nunca abordaron la causa raíz en `load_conversation()`
6. **Definición duplicada** de `SLIDING_WINDOW = 20` en líneas 45 y 97 creaba un peligro de mantenimiento

**Causa raíz en español accesible:**
Cada vez que Fernando hablaba con el bot, el sistema guardaba hasta 20 mensajes en un archivo. Pero cuando cargaba el historial para enviarlo a la IA de MiniMax, **no recortaba** ese historial a 20 mensajes. Así que si había 50 mensajes guardados, los 50 se enviaban a MiniMax. La IA de MiniMax tiene un límite de cuánto texto puede procesar de una vez — cuando el historial se hizo muy grande, dejó de funcionar.

---

## 4. Qué se Corrigió

### Cambio 1: Eliminar constante duplicada
Se eliminó la definición duplicada de `SLIDING_WINDOW = 20` en la línea 97 de `brain.py`. Solo queda la definición canónica en la línea 45. Esto previene que futuras ediciones a una línea no se propaguen a la otra.

### Cambio 2: Observabilidad de truncamiento
Se añadió un log `CRITICAL` en `load_conversation()` cuando el truncamiento se dispara:
```python
logger.critical(f"[load_conversation] Truncating {len(history)} messages to SLIDING_WINDOW={SLIDING_WINDOW} for user {user_id}")
```
Antes, el truncamiento ocurría silenciosamente. Ahora, cada vez que el historial excede 20 mensajes, el sistema escribe un log CRITICAL que el monitor proactivo puede detectar inmediatamente.

### Cambio 3: Validación de inicio `_validate_conversation_store_size()`
Nueva función de validación al iniciar el contenedor que lee `conversation_store.json` y lanza CRITICAL si algún usuario tiene más de 30 mensajes (1.5 × 20). Esto detecta la acumulación antes de que cause fallos en la API de MiniMax.

### Cambio 4: Suite de pruebas
Se creó `/root/minimax-agent/tests/test_conversation_store.py` con 11 pruebas cubriendo:
- `load_conversation()` trunca correctamente cuando excede la ventana
- `load_conversation()` retorna historial completo cuando está dentro de la ventana
- `load_conversation()` retorna lista vacía cuando no existe archivo
- `_validate_conversation_store_size()` retorna True cuando todos ≤ 30 mensajes
- `_validate_conversation_store_size()` retorna False cuando algún usuario > 30 mensajes
- Manejo de JSON corrupto (no-crash)
- Verificación de definición única de SLIDING_WINDOW

**Resultado:** 11/11 pruebas PASS en 0.16s

---

## 5. Mejoras Estructurales Añadidas

1. **Eliminación de deuda técnica:** La definición duplicada de `SLIDING_WINDOW` era un residuo de desarrollo inicial que ningún linter había detectado. Ya no puede causar confusión.

2. **Observabilidad completa:** El log CRITICAL en truncamiento hace que el patrón de "conversación crece sin límite" sea inmediatamente visible en los logs del contenedor, sin esperar a que MiniMax retorne el error 2013.

3. **Validación proactiva:** `_validate_conversation_store_size()` corre al iniciar el contenedor. Si el archivo de conversación tiene más de 30 mensajes para cualquier usuario, un CRITICAL se emite inmediatamente — antes de que Fernando experimente un fallo.

4. **Prevención de recurrencia:** La combinación de (a) log CRITICAL cuando truncamiento ocurre en producción y (b) validación de inicio que detecta acumulación antes de que cause errores, hace que esta clase de error sea imposible de ignorar o pasar desapercibida.

## Resumen

El problema era que el historial de conversación del bot de Telegram crecía sin límite porque `load_conversation()` solo recortaba mensajes al guardar, no al cargar. Después de 25 repeticiones del mismo error con 0% de éxito en las correcciones, el pipeline identificó la causa raíz real: el truncamiento ocurría en la capa incorrecta (save vs. load). La corrección agregó: (1) log CRITICAL cuando el truncamiento ocurre, (2) validación al inicio que detecta acumulación excesiva, (3) eliminación de la definición duplicada de la constante. Container healthy, 11/11 tests passing, zero errores nuevos. Los issues pre-existentes de Odoo (HTTP 500, 409, ReadTimeout) son problemas separados del backend que deben investigarse independientemente.
