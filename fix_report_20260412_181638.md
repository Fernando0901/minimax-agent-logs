# Fix Report — agent_chat startup probe timeout false positives

## 1. Problema Reportado

El monitor proactivo de salud detectó 4 errores recurrentes del tipo `CONNECTION_ERROR — ReadTimeout` en el endpoint `/agent/chat` del microservicio Odoo (`odoo-python-agent:8000`), generados por la validación de inicio del sistema `minimax-agent`. Los errores occurred en:
- `2026-04-12 17:45:45,237` — WARNING: `startup check agent_chat: CONNECTION_ERROR — ReadTimeout`
- Enlaces a Mozilla Developer Network para errores HTTP 500 y 409 también presentes

El análisis de tendencias reveló degradación severa del tiempo de respuesta: de ~5 segundos a 114,680 ms (114 segundos) para las 17:57. Este patrón de errores se había repetido más de 25 veces con una tasa de éxito histórico del 0% en la resolución.

## 3. Causa Raíz

**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Función:** `_validate_odoo_bridge_endpoints()`
**Línea:** ~514

**Mecanismo técnico:** La función `_validate_odoo_bridge_endpoints()` utilizaba una única llamada probe con `httpx.AsyncClient(timeout=5)` — un timeout total de 5 segundos (connect + read combinados) **sin reintentos** — para verificar TODOS los endpoints del microservicio Odoo, incluyendo `/agent/chat`.

Cuando el backend de Odoo está bajo carga, el endpoint `/agent/chat` necesita **30-169 segundos** para responder (operaciones XML-RPC documentadas en sesiones previas 20260412_164522). El probe de 5 segundos agotaba su timeout en el primer intento mientras que la función de producción `agent_chat()` (que usa `_DEFAULT_TIMEOUT=120s` + `_retry_with_backoff()` con 2 reintentos) habría succeed. Esto generaba **falsos positivos en cascada**: el probe fallido activaba alertas CRITICAL, creando un patrón de 25+ recurrencias con 0% de resolución histórica.

El mismatch específico era:
- **Probe de startup:** `timeout=5`, sin reintentos
- **Comportamiento de producción:** `_DEFAULT_TIMEOUT=120s` + 2 reintentos con delays exponenciales (2s, 4s)

La causa raíz no era el endpoint `/agent/chat` en sí (que existía y respondía 200 OK), sino el **timeout demasiado corto del probe de validación de startup** combinado con la ausencia de lógica de reintentos.

## 4. Qué se Corrigió

Se implementaron 3 cambios estructurales en `/root/minimax-agent/agent/odoo_bridge.py`:

### Cambio 1: Nueva función `_probe_agent_chat_with_backoff()`
Función helper asíncrona que hace probe del endpoint `/agent/chat` con:
- **Timeout por intento:** 30 segundos
- **Reintentos:** 2 intentos (delays: 2s, 4s)
- **Comportamiento:** Retorna `True` en caso de éxito, `False` tras 3 intentos agotados
- **Errores HTTP 422/500:** Retorna `False` sin elevar excepción
- Refleja exactamente el comportamiento de producción de `agent_chat()`

### Cambio 2: Nueva validación `_validate_agent_chat_probe_config()`
Check de startup basado en AST que:
- Lee el código fuente de `odoo_bridge.py`
- Verifica que `_probe_agent_chat_with_backoff` tenga timeout >= 30s y retry count >= 2
- Si se viola: eleva `AssertionError` con mensaje `AGENT_CHAT_PROBE_VALIDATION FAILED`
- **Propósito:** Prevención de recurrencia — si alguien en el futuro reduce el timeout, el contenedor falla rápido con CRITICAL visible

### Cambio 3: Actualización del loop de validación
En `_validate_odoo_bridge_endpoints()`, el loop ahora:
- Detecta si `name == "agent_chat"`
- Llama a `_probe_agent_chat_with_backoff()` en lugar del probe genérico de 5s
- Otros endpoints conservan el timeout de 5s (intencionado — checks ligeros deben fallar rápido)

### Cambio 4: Suite de tests
Archivo `tests/test_agent_chat_probe.py` creado con 8 tests cubriendo:
- Happy path (200 OK)
- ReadTimeout retry → éxito en 2do intento
- ReadTimeout exhaust → False tras 3 fallidos
- ConnectError retry → éxito en 2do intento
- HTTP 422/500 → retorna False sin raise
- Validación passthrough y fail conditions

## 5. Mejoras Estructurales Añadidas

### Prevención de Recurrencia
- El probe de `/agent/chat` ahora usa timeout de 30s + 2 reintentos, **igual que la llamada de producción**
- Si el backend de Odoo es genuinamente lento (30-90s), los reintentos absorben la latencia
- Solo si el endpoint está genuinamente caído por >90s se registra `CONNECTION_ERROR` (comportamiento correcto, no falso positivo)

### Capa de Observabilidad
Logs estructurados añadidos:
- `startup check agent_chat: OK` — éxito (sin cambios)
- `startup check agent_chat: ReadTimeout on attempt N/3` — muestra en qué intento ocurrió timeout
- `startup check agent_chat: ConnectError on attempt N/3` — muestra error de conexión por intento
- `startup check agent_chat: CONNECTION_ERROR after 3 attempts` — falla final tras reintentos agotados
- `AGENT_CHAT_PROBE_VALIDATION: PASSED (timeout=30s, retries=2)` — confirmación de validación de startup

### Tests Automatizados
- 8 tests en `tests/test_agent_chat_probe.py`
- Verifican: retry logic, timeout handling, HTTP error handling, validation pass/fail
