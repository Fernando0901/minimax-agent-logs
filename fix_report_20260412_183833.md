## 3. Causa Raíz

### Versión accessible para Fernando

**El problema tenía dos componentes:**

**Componente 1 — Historial de conversación demasiado grande:**
El agente guarda los últimos 20 mensajes de cada conversación (constante `SLIDING_WINDOW = 20`). Sin embargo, cuando el agente hace llamadas a herramientas (buscar en Odoo, hacer búsquedas web, generar imágenes), cada resultado de herramienta se añade a la lista de mensajes que se envía a MiniMax. Estos mensajes adicionales NO estaban sujetos al límite de 20 mensajes. Si una conversación tenía 3 iteraciones de tool-calls, el contexto real enviado a MiniMax era de ~27+ mensajes, excediendo el límite del modelo y causando que devolviera `choices: None`.

**Componente 2 — Sin protección contra respuestas lentas:**
Cuando MiniMax se volvía lento (114 segundos observados), el código esperaba hasta 120 segundos (timeout de httpx) antes de dar timeout. Cuando eso pasaba, el agente intentaba un retry que también se colgaba por otros 114 segundos. El resultado era que Fernando esperaba ~4 minutos sin respuesta.

**Nota importante:** La sesión anterior `20260412_175610` intentó arreglar este problema pero solo añadió "observabilidad" (logs cuando el truncamiento ocurría) — NO redujo el tamaño del historial ni añadió protección contra timeouts. Por eso el problema volvió a ocurrir 13 minutos después.

### Technical Detail

**Archivo:** `/root/minimax-agent/agent/brain.py`
**Funciones:** `load_conversation()` (lineas 284-293) + `chat()` (lineas 1317-1506) + `call_minimax()` (lineas 405-475)

**Mecanismo exacto:**
1. `SLIDING_WINDOW = 20` en líneas 51 y 103 (duplicado) — demasiado grande para contexto efectivo considerando tool-call loops
2. Tool call loop en `chat()` (lineas 1343-1499) acumula mensajes `assistant` + `tool result` que NO pasan por la truncación de `load_conversation()`
3. httpx `Timeout(read=120.0)` — demasiado próximo al límite de MiniMax; 114s de respuesta activa el timeout
4. Retry logic para `choices: None` usa el mismo timeout de 120s — retry también se cuelga
5. No existe `asyncio.wait_for()` con deadline absoluto envolviendo la cadena `chat()` → `call_minimax()`

**Evidencia clave:**
- Spike de 114,680ms a las 17:57:46 — justo debajo del límite de 120s
- Conversación del usuario 8288612046 (Fernando) tenía exactamente 20 mensajes — al límite del SLIDING_WINDOW
- Commit `1b5227c` ("remove agent timeouts — increase all baselines to 6h") potencialmente eliminó protecciones de timeout previamente existentes
- 25 recurrencias previas con 0% de tasa de éxito — todas las fix anteriores abordaban la capa incorrecta

## 4. Qué se Corrigió

### Cambio 1 — SLIDING_WINDOW reducido de 20 a 10
**Archivo:** `/root/minimax-agent/agent/brain.py`
**Líneas:** 51 y 103 (ambas definiciones duplicadas)

Reducir el historial de 20 a 10 mensajes corta a la mitad el contexto base, dejando espacio para hasta 5+ iteraciones de tool-call sin exceder los límites del modelo.

```
# Antes: SLIDING_WINDOW = 20
# Después: SLIDING_WINDOW = 10
```

### Cambio 2 — asyncio.wait_for(timeout=60.0) en todos los sitios de call_minimax
**Archivo:** `/root/minimax-agent/agent/brain.py`
**Líneas:** 3 sitios dentro de `chat()` (línea ~1398, ~1469, ~1539)

Cada llamada a `call_minimax()` dentro de `chat()` ahora está envuelta con `asyncio.wait_for(..., timeout=60.0)` con try/except que captura `TimeoutError` y devuelve un mensaje de error graceful. Esto impone un deadline absoluto de 60 segundos para toda la operación lógica, previniendo hangs catastróficos.

```
# Antes: data = await call_minimax(...)
# Después:
try:
    data = await asyncio.wait_for(call_minimax(...), timeout=60.0)
except TimeoutError:
    return {"error": "La solicitud a MiniMax tardó demasiado tiempo..."}
```

### Cambio 3 — Validación AST de startup `_validate_sliding_window_and_timeout()`
**Archivo:** `/root/minimax-agent/agent/brain.py`
**Línea:** ~1113

Nueva función de validación que:
1. Verifica que `SLIDING_WINDOW <= 10`
2. Verifica que `asyncio.wait_for` está presente en la cadena de llamadas a `call_minimax`
3. Verifica que el timeout passado sea 60.0s

Si cualquier verificación falla, el contenedor lanza `CRITICAL` en startup y no arranca normalmente. Esto es una protección anti-regresión.

### Cambio 4 — Suite de tests `test_minimax_timeout_and_context.py`
**Archivo:** `/root/minimax-agent/tests/test_minimax_timeout_and_context.py`
**Tests:** 5/5 pasando

- `test_sliding_window_value` — verifica SLIDING_WINDOW == 10
- `test_asyncio_wait_for_present_in_chat` — verificación AST de wait_for envolviendo call_minimax
- `test_wait_for_timeout_is_60` — verificación AST de timeout == 60.0
- `test_load_conversation_truncates_at_load` — truncación funciona correctamente
- `test_validate_sliding_window_and_timeout` — startup check pasa correctamente

## 5. Mejoras Estructurales Añadidas

### Prevención de Recurrencia
- **SLIDING_WINDOW 20→10:** Reduce el contexto base a la mitad. Con tool-call loops generando ~3-5 mensajes adicionales por iteración, el contexto efectivo ahora es ~18 mensajes en lugar de ~27+, manteniendo el presupuesto de MiniMax-M2.7 dentro de límites seguros.
- **Deadline absoluto de 60s:** Cada solicitud lógica a MiniMax (incluyendo todos los retries) tiene un deadline absoluto. Si MiniMax no responde en 60 segundos, se devuelve error graceful en lugar de hang de 4 minutos.

### Capa de Validación
- **`_validate_sliding_window_and_timeout()` AST startup check:** Fallos rápidos (fail-fast) si SLIDING_WINDOW > 10 o si `asyncio.wait_for` falta de la cadena de llamadas. El contenedor no arranca si la validación falla.

### Observabilidad
- Logs estructurados en cada sitio de `asyncio.wait_for` con etiqueta `[call_minimax] Absolute deadline exceeded — raising TimeoutError` cuando el timeout de 60s dispara.
- Logs con `[call_minimax] Timeout value: 60.0s` en entrada de call_minimax para trazabilidad.
