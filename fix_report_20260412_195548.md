## 3. Causa Raíz

### Causa Raíz (Español — para Fernando)

El sistema de monitoreo de salud del contenedor (`healthcheck.sh`) utiliza el comando `pgrep` para verificar que el proceso principal del bot está corriendo. Sin embargo, el comando `pgrep` pertenece al paquete sistema `procps`, y este paquete **no estaba instalado** en el contenedor Docker.

Cuando el script de healthcheck intentaba ejecutar `pgrep -f "agent.main"`, el shell retornaba **exit code 127** ("command not found"), lo cual hacía que el healthcheck fallara y Docker marcara el contenedor como **UNHEALTHY**.

Este problema ocurrió porque el script `healthcheck.sh` fue introducido en la sesión 20260412_185955, pero el `Dockerfile` nunca fue actualizado para incluir la dependencia `procps` en la lista de paquetes a instalar.

El monitoreo proactivo detectaba el estado UNHEALTHY del contenedor y generaba alertas, confundiendo este problema de infraestructura con errores reales de conexión a Odoo.

### Technical Root Cause (English)

**File:** `/root/minimax-agent/Dockerfile` (line 10)
**Mechanism:** The Dockerfile's `apt-get install` command was missing the `procps` package, which provides the `pgrep` binary. When `healthcheck.sh` CHECK 3 executed `pgrep -f "agent.main"`, the shell returned exit code 127 (command not found). The healthcheck script then exited with code 1, causing Docker's healthcheck mechanism to fail continuously with a `FailingStreak`. The `proactive_health_monitor` detected the blank/unhealthy container status and created false `CONTAINER_UNHEALTHY` task files, while also counting Odoo startup validation errors (HTTP 500s from stale CRM lead IDs) as part of the same failure pattern. This broke the health monitoring subsystem that was supposed to detect real problems.

**Confidence:** 0.95
**Evidence:**
1. Dockerfile line 10 had `procps` absent (added in session 20260412_193251, commit `85cb970`)
2. Container rebuild at 19:48:05 UTC → `HEALTHCHECK_PASSED` at 19:50:44 (within 2 minutes)
3. `pgrep` exit code 127 confirmed by `_PGREP_EXIT` handling in healthcheck.sh
4. Docker inspect showed `FailingStreak: 0` and health status `healthy` at 19:56
5. The 26 recurrence count + 0% historical success rate was explained by this gap

## 4. Qué se Corrigió

### Corrección Ya Aplicada (sesión 20260412_193251)

El fix primario fue aplicado en una sesión anterior (20260412_193251):

1. **Paquete `procps` añadido al Dockerfile**: La línea 10 del Dockerfile ahora incluye `procps` en el comando `apt-get install`, asegurando que `pgrep` esté disponible en el contenedor.

2. **`_validate_healthcheck_dependencies()` añadido** en `main.py`: Esta función de validación durante el startup usa `shutil.which("pgrep")` para verificar que pgrep existe antes de que el contenedor empiece a servir requests. Si falta, el contenedor falla con un mensaje CRITICAL descriptivo.

3. **CHECK 3 reescrito con detección de exit code 127**: El script `healthcheck.sh` ahora detecta explícitamente cuando `pgrep` retorna exit code 127 y loguea un mensaje descriptivo: `"HEALTHCHECK_FAILED: pgrep not found — container missing procps package"`.

4. **Patrón de búsqueda corregido**: El patrón grep cambió de `python.*main.py` a `agent.main` para correctamente encontrar procesos iniciados con `python -m agent.main`.

5. **Suite de tests creada**: `tests/test_healthcheck_dependencies.py` con 5 tests que verifican:
   - `shutil.which("pgrep")` returns non-None
   - pgrep exit code 0 para el proceso `agent.main`
   - healthcheck.sh exit 0 cuando todos los checks pasan
   - healthcheck.sh exit 1 con mensaje descriptivo cuando pgrep falta
   - Startup check `_validate_healthcheck_dependencies()` pasa en el contenedor actual

## 5. Mejoras Estructurales Añadidas

### Para Prevenir Recurrencia

1. **Validación de dependencias en startup**: `_validate_healthcheck_dependencies()` en `main.py` verifica que `pgrep` existe antes de que el contenedor acepte tráfico. Si `procps` falta, el contenedor falla inmediatamente con un mensaje claro.

2. **Deteción de exit code 127 en healthcheck.sh**: El script ahora detecta específicamente cuando `pgrep` no existe (exit 127) y provee un mensaje de error que indica exactamente qué paquete falta.

3. **Logs estructurados**:
   - `HEALTHCHECK_PASSED` cada 30s cuando el healthcheck succeed
   - `HEALTHCHECK_FAILED: [razón]` cuando falla
   - `PGREP_DEPENDENCY_CHECK: PASSED` en logs de startup

4. **Test suite automatizada**: `tests/test_healthcheck_dependencies.py` con 5 tests que hubieran detectado este problema automáticamente.

### Observabilidad

- El healthcheck.sh ahora retorna exit code 1 con razones específicas:
  - `pgrep command not found (procps package missing)` — indica gap en Dockerfile
  - `No agent.main process found` — indica crash de aplicación
  - `log stale — {age}s old` — indica starvation del event loop

- El threshold de Docker para UNHEALTHY es automático después de fallos consecutivos. El `proactive_health_monitor` también crea task files cuando el estado del contenedor está en blanco o no saludable.
