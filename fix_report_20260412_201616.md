## 3. Causa Raíz

### Explicación del estado "****"

El estado `****` reportado por Docker no es un estado de salud válido de Docker — es un **artefacto de visualización**. Cuando un script de healthcheck en bash produce output con caracteres no-imprimibles o de control (por ejemplo, desde un path de error que sale tempranamente), Docker trunca/transforma este output en caracteres `*` en su pantalla de estado.

### Causa subyacente (ya resuelta)

El script de healthcheck `healthcheck.sh` producía output no-válido durante el estado de falla **antes** de que se instalara el paquete `procps` en el Dockerfile (sesión 20260412_193251). Específicamente:
- El CHECK 3 usaba `pgrep -f "python.*main.py"` pero `pgrep` no existía en el contenedor sin `procps`
- El script producía output con caracteres de control que Docker convertía a `****`
- El contenedor fue reconstruido a las 19:48:05 UTC con TODAS las correcciones acumulativas aplicadas

## 5. Mejoras Estructurales Añadidas

Las siguientes mejoras estructurales fueron aplicadas en sesiones previas (no esta sesión — no se requirieron cambios nuevos):

### Validación al startup
- `_validate_healthcheck_dependencies()` — verifica que `pgrep` exista vía `shutil.which`
- `_validate_healthcheck_script()` — verifica que `healthcheck.sh` exista y sea ejecutable
- `_validate_odoo_bridge_endpoints()` — probes por-endpoint con payloads válidos
- `_validate_connection_pool_max_wait()` y `_validate_odoo_bridge_timeout_values()`

### Observabilidad
- `healthcheck.log` en `/root/minimax-agent/logs/healthcheck.log` — registra cada healthcheck con HEALTHCHECK_PASSED/FAILED + timestamp
- Container health visible vía `docker inspect minimax-agent | grep -A 10 Health`
- 24+ entradas HEALTHCHECK_PASSED confirmadas de 20:05 a 20:22 UTC

### Pattern de auto-recuperación identificado
El estado `****` como artifacto de Docker (output no-válido del healthcheck) indica que el gap entre la introducción del script `healthcheck.sh` (sesión 20260412_185955) y la actualización del Dockerfile para incluir `procps` (sesión 20260412_193251) fue la causa raíz del problema. Este gap de 26+ recurrencias ahora está documentado como patrón conocido en `pipeline/CLAUDE.md`.
