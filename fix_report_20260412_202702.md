# Fix Report — Startup probe retry mismatch causing false CRITICAL alerts

## 1. Problema Reportado

El monitor de salud proactivo detectó que el contenedor `minimax-agent` reportaba estado `UNHEALTHY` con el estado "****". Este era un problema recurrente con **21+ errores de CONNECTION_ERROR tipo ReadTimeout** en las últimas 2 horas, donde el 58% de los endpoints del microservicio Odoo (7 de 12) fallaban la validación de startup.

El problema se manifestaba como:
- `CONTAINER_UNHEALTHY` — estado del contenedor truncado a "****"
- `update_opportunity: CONNECTION_ERROR — ReadTimeout` — 21+ ocurrencias
- `STARTUP VALIDATION FAILED — 58% endpoints down (7/12)` — alerta CRITICAL
- Degradación de respuesta de 5s → 114s en 8 minutos

Fernando también reportó que el agente no respondía mensajes de Telegram con "La respuesta tomó demasiado tiempo".

## 3. Causa Raíz

### Causa técnica
**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Función:** `_validate_odoo_bridge_endpoints()` (líneas 655-672)
**Mecanismo:** Las sondas de startup para los 11 endpoints no-`agent_chat` usaban `timeout=5` sin reintentos, mientras que el código de producción para los mismos endpoints usa `httpx.Timeout(120.0) + _retry_with_backoff()` con 2 reintentos y backoff exponencial (2s/4s). Bajo carga de Odoo (30-169s observados), las sondas de startup agotaban su timeout de 5s antes de que Odoo pudiera responder, produciendo `ReadTimeout` falsos clasificados como `CONNECTION_ERROR` y-contados hacia el threshold del 58%.

Adicionalmente, `opportunity_id=1` en la sonda de `update_opportunity` no existe en Odoo, produciendo HTTP 500 legítimo (error de aplicación) que era malclasificado como falla de infraestructura.

El patrón correcto ya existía en `_probe_agent_chat_with_backoff()` (líneas 463-518, añadido en sesión 20260412_181638) pero solo se había aplicado al endpoint `agent_chat`, no a los otros 11 endpoints.

### Explicación accesible para Fernando
El agente tiene un "chequeo de salud" que verifica que el servicio de Odoo esté funcionando cada vez que arranca. Ese chequeo usaba un timeout muy corto (5 segundos) y sin reintentos, pero el servicio de Odoo a veces tarda hasta 2-3 minutos bajo carga. Cuando el chequeo se agotaba (timeout), el sistema pensaba que el servicio estaba caído y mandaba alertas de CRITICAL. Además, algunos IDs de prueba que no existen en Odoo producían errores HTTP 500 que también se contaban como "caídas".

## 4. Qué se Corrigió

### Código modificado (3 archivos)

**`/root/minimax-agent/agent/odoo_bridge.py`:**
- Añadida función `_probe_endpoint_with_backoff()` generalizada con 30s timeout + 2 reintentos (2s/4s delays) — misma lógica que `_probe_agent_chat_with_backoff()` pero para todos los endpoints
- Reemplazado el loop de sondas planas (`timeout=5`) con llamadas a `_probe_endpoint_with_backoff()` para los 11 endpoints no-`agent_chat`
- HTTP 400/409/422/500 ahora clasificados como `APPLICATION_ERROR` (no cuentan como falla de infraestructura)
- Solo `CONNECTION_ERROR` (ReadTimeout, ConnectError, HTTP 503 después de reintentos) cuenta como falla
- Añadida función `_validate_startup_probe_retry_config()` — AST startup check que verifica que todas las sondas usen timeout ≥ 30s y tengan lógica de reintento
- Añadida wired en `main.py` durante startup

**`/root/minimax-agent/agent/main.py`:**
- Añadida llamada a `odoo_bridge_module._validate_startup_probe_retry_config()` en la secuencia de startup después de `_validate_odoo_bridge_endpoints()`

**`/root/minimax-agent/tests/test_odoo_bridge_startup_probe_retry.py` (nuevo):**
- Suite de 10 tests cubriendo: éxito en HTTP 200, reintento en ReadTimeout, clasificación de errores HTTP 400/409/422/500/503 como APPLICATION_ERROR vs CONNECTION_ERROR, AST validation pass/fail

| Layer | Improvement |

| **Error Classification** | Distinción clara entre APPLICATION_ERROR (HTTP 4xx/5xx de datos inválidos de Odoo, no cuenta) vs CONNECTION_ERROR (fallas de red reales, sí cuenta) |
