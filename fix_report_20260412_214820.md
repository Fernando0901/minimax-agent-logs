## 3. Causa Raíz

### Explicación accesible (en español)

El sistema tiene dos "relojes" para medir si el servicio de Odoo responde bien:

1. **El reloj de producción** (el que usa el agente cuando trabaja): tiene 120 segundos de timeout + 2 reintentos
2. **El reloj de la sonda de startup** (el que verifica que todo esté bien cuando el contenedor inicia): tenía solo 30 segundos

Cuando Odoo está ligeramente lento (responde entre 30 y 120 segundos), el reloj de producción habría esperado y habría succeed, pero el reloj de la sonda se agotaba a los 30 segundos y reportaba un error falso de "conexión" — aunque el servicio en realidad estaba funcionando.

### Detalle técnico

- **Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
- **Función problematica:** `_probe_endpoint_with_backoff()` (línea 584)
- **Línea exacta:** 598: `probe_timeout = httpx.Timeout(30.0, connect=10.0, read=30.0)`
- **Producción:** `_DEFAULT_TIMEOUT = httpx.Timeout(120.0, connect=10.0, read=120.0)` + `_retry_with_backoff()` con 2 reintentos (delays de 2s/4s)
- **Gap:** La sonda agota 3 intentos × 30s = 90s total; producción tiene ~126s antes de agotar todos los reintentos
- **Clasificación errónea:** ReadTimeout se contaba como `connection_error` (fracaso de infraestructura) cuando en realidad era solo lentitud del backend — Odoo responding between 30-120s
- **Umbral CRITICAL:** Si >50% de las sondas fallan como `connection_error`, se dispara alerta CRITICAL. Un solo falso `connection_error` en `update_opportunity` combinado con otros podía empujar el porcentaje sobre el umbral
- **Historia:** 28 recurrencias acumuladas en múltiples sesiones con 0% de tasa de éxito — nunca se había ataca la causa raíz correcta

## 4. Qué se Corrigió

Se implementaron 4 cambios estructurales en `/root/minimax-agent/agent/odoo_bridge.py`:

### Cambio 1 — Nueva función `_probe_slow_endpoint_with_backoff()`
- Timeout de lectura de 120 segundos (`httpx.Timeout(120.0, connect=10.0, read=120.0)`)
- 2 reintentos con delays de 2s y 4s (espejo exacto del cliente de producción)
- Mismo formato de retorno 3-tupla que la función original
- Sin modificar `_probe_endpoint_with_backoff()` — todas las demás sondas conservan su comportamiento de 30s

### Cambio 2 — Enrutamiento de `update_opportunity` a la nueva sonda
- En el loop de `_validate_odoo_bridge_endpoints()`, se agregó un branch condicional para `update_opportunity`
- Ahora llama a `_probe_slow_endpoint_with_backoff()` en lugar de `_probe_endpoint_with_backoff()`
- Otras sondas (search, get_product_price, etc.) siguen usando el timeout de 30s

### Cambio 3 — Check AST de startup `_validate_update_opportunity_probe_config()`
- Verifica que `_probe_slow_endpoint_with_backoff()` exista en el módulo
- Verifica que el timeout de lectura sea `>= 120` segundos
- Verifica que `max_retries >= 2`
- Fails fast con `AssertionError` si la configuración es incorrecta
- Usa verificación AST (`ast.Constant`) para leer el código fuente directamente

### Cambio 4 — Suite de tests (10 tests)
- Archivo: `tests/test_update_opportunity_probe_timeout.py`
- 3 tests para `_validate_update_opportunity_probe_config()`: pasa cuando la config es correcta, falla cuando timeout < 120 o retries < 2
- 7 tests para `_probe_slow_endpoint_with_backoff()`: HTTP 200/400/409/422/500, ReadTimeout tras todos los retries, retry exitoso en segundo intento
- **Resultado: 10/10 PASS**

## 5. Mejoras Estructurales Añadidas

### Prevención de recurrencia
El check AST `_validate_update_opportunity_probe_config()` asegura que cualquier cambio futuro que reintroduzca un timeout de 30s para `update_opportunity` será detectado inmediatamente — el contenedor fallará al iniciar con un `AssertionError` claro antes de llegar a producción.

### Observabilidad
La nueva función de sonda tiene logging estructurado para reintentos:
- `ReadTimeout on attempt X/3, retrying in Ys` — ahora fires para el umbral correcto (120s, no 30s)
- `CONNECTION_ERROR — ReadTimeout after 3 attempts` — ahora solo cuando realmente está inalcanzable (no cuando está lento)

### Capa de validación
La validación de startup ahora distingue claramente entre:
- `connection_error` = Odoo realmente no responde ( Contección rechazada, DNS fail, etc.)
- `application_error` = Odoo responde pero con error HTTP (400, 409, 422, 500) — estos NO se cuentan para el umbral CRITICAL
