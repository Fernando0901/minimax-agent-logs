## 1. Problema Reportado

El estado del contenedor se reportaba como `****` (estado vacío/truncado) en la salida de `docker ps`, lo cual activaba la creación de archivos de tarea `proactive_container_{timestamp}.md` con clasificación `CONTAINER_UNHEALTHY`.

Este problema había ocurrido en 6+ sesiones anteriores (20260412_181611, 184806, 185621, 191614, 195334, 201616) con 0% de tasa de éxito en la resolución — el contenedor siempre estaba saludable en el momento de la verificación manual, y el problema se autcorreggía después de que el contenedor completaba su arranque.

## 3. Causa Raíz

### Causa Raíz (Español)
Cuando el contenedor `minimax-agent` se reinicia o está en fase de arranque, Docker devuelve una cadena vacía (`""`) en el campo `.State.Health.Status` en lugar de `"starting"`, `"unhealthy"` o `"healthy"`. El monitor proactivo ejecutaba `docker inspect --format "{{.State.Health.Status}}"` y asignaba el resultado a `status = result.stdout.strip()`. Si `status == ""`, la comparación `status != "healthy"` era `True`, lo que inmediatamente creaba un archivo de tarea `CONTAINER_UNHEALTHY`.

La visualización "****" en `docker ps` es un artefacto conocido de Docker: cuando el campo de estado de salud está vacío o contiene caracteres no imprimibles durante el reinicio del contenedor, Docker trunca la salida a "****". Es cosmético y refleja la naturaleza transitoria del estado durante el reinicio, no un defecto real.

El contenedor **siempre estuvo saludable** durante estas alertas: el script de healthcheck (`/root/minimax-agent/scripts/healthcheck.sh`) implementa 3 sondas independientes (Telegram getMe API, frescura de log a 300s, pgrep de agent.main) y todas pasaban correctamente.

### Detalle Técnico
- **Archivo:** `/root/vision/proactive_monitor.py`
- **Función:** `_check_container_health()` — líneas 264-342
- **Línea exacta del defecto:** 286 (`if status != "healthy":` — trata cadena vacía como no-saludable)
- **Llamadas externas:** `docker inspect` subprocess — timeout 12s
- **Manejo de errores:** Parcial — captura excepciones de subprocess pero trata estado vacío como enfermizo
- **Confidence:** 0.92

### Línea Temporal Reconstruida
1. Contenedor **reiniciado a las 22:01:07 UTC** (docker StartedAt timestamp)
2. Sesión anterior del pipeline remountó `/usr/bin/docker` para acceso independiente de PATH
3. El script healthcheck necesita ~6 minutos de start_period (intervalo 30s × 3 retries × start_period 30s) antes de pasar
4. Scan proactivo ejecutado a las **22:07:58 UTC**
5. A las 22:07:58, el contenedor tenía ~7 minutos pero aún podía estar en ventana de arranque
6. `docker inspect` devolvió estado vacío → `container_ok=False` → tarea creada

## 4. Qué se Corrigió

### Modificación Principal — `_check_container_health()` en `/root/vision/proactive_monitor.py`

Se implementó una **evaluación de tres niveles** en lugar de la comparación simple `status != "healthy"`:

| Estado de Docker | Comportamiento Anterior | Comportamiento Nuevo |
|-----------------|------------------------|---------------------|
| `status == "healthy"` | Retorna True, OK | Retorna True, "healthy" ✅ (sin cambio) |
| `status == ""` (vacío) | Retorna False, crea tarea CONTAINER_UNHEALTHY ❌ | Retorna True, "transient startup" ✅ (nuevo) |
| `status` no vacío y no healthy | Retorna False, crea tarea CONTAINER_UNHEALTHY ✅ | Sin cambio — comportamiento correcto preservado |

**Cambios en el código:**
- **Líneas 291-296 (nuevo):** Empty string → log WARNING + retorna `(True, "transient startup")` — no se crea archivo de tarea
- **Líneas 298-300 (nuevo):** Explicit "healthy" check → retorna `(True, "healthy")`
- **Líneas 302-321:** Non-healthy non-empty → crea tarea si es necesario (lógica original preservada)

### Validación de Inicio Añadida
- **Función:** `_validate_empty_status_handling()` — líneas 371-394
- **Propósito:** Auto-verificación del sistema de tres niveles en tiempo de importación del módulo
- **Ejecución:** `asyncio.run(_validate_empty_status_handling())` al cargar el módulo
- **Comportamiento en fallo:** `log.critical()` + `raise RuntimeError` — fail-fast antes de que el monitoreo productivo inicio

### Función Auxiliar Añadida
- **`_classify_status()` — línea 360:** Función helper para validación, clasifica vacío como "transient"

### Logs Estructurados Añadidos
- `log.warning(f"[proactive] container health status empty (transient during startup/restart) — skipping alert")` — cuando estado vacío detectado
- `log.info(f"[proactive] container health check passed: status={status}")` — cuando estado es healthy

## Resumen

El fix de esta sesión resuelve el problema de alertas falsas de CONTAINER_UNHEALTHY que ocurrió en 6+ sesiones anteriores. La causa raíz era que `_check_container_health()` en `/root/vision/proactive_monitor.py` trataba el estado de salud vacío de Docker (durante reinicio/transición del contenedor) como enfermizo. La solución implementa una evaluación de tres niveles que distingue entre estado transitorio vacío (advertencia, sin tarea), estado healthy (OK), y estado genuinamente no-saludable (crear tarea).

El contenedor minimax-agent está saludable, los 10 tests pasan, y el patrón de falla ahora está documentado en `pipeline/CLAUDE.md` para recurrencia futura.
