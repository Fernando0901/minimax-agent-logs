## 3. Causa Raíz

**Mecanismo técnico:** `SLIDING_WINDOW=10` genera un historial de ~500 tokens de mensajes, que combinado con la sobrecarga fija del system prompt (~4000 tokens) y las definiciones de herramientas (~3000 tokens) enviadas en cada llamada a la API de MiniMax, totaliza ~7500 tokens. Esto se aproxima o excede el límite de la ventana de contexto del modelo MiniMax-M2.7, causando errores intermitentes 2013 "context window exceeds limit".

**Problema secundario identificado:** Cuando `call_minimax` recibe una respuesta 2013, el retry existente (para `choices: None`) reenvía la **misma solicitud con contexto idéntico**, garantizando que el retry también falle. No había manejo específico para el código de error 2013.

**Contexto histórico de la recurrencia:**
- Session 20260412_175610 agregó observabilidad (CRITICAL log cuando el truncamiento fires) pero NO prevención de recurrencia
- Session 20260412_183833 redujo SLIDING_WINDOW de 20 a 10 y agregó `asyncio.wait_for(timeout=60.0)` para prevenir el timeout de 114s, pero no resolvió el agotamiento de la ventana de contexto (2013)
- Los 28 recurrencias con 0% de tasa de éxito indican que los fixes previos atacaron la capa equivocada

**Detalles de las definiciones duplicadas de SLIDING_WINDOW:**
- Línea 51: `SLIDING_WINDOW = 10` (primera definición)
- Línea 103: `SLIDING_WINDOW = 10` (segunda definición, hazard de mantenimiento)

## 4. Qué se Corrigió

**Paso 1:** Reducido `SLIDING_WINDOW` de 10 a 5 en la primera definición (línea 51)
- Impacto: reduce el historial de mensajes de ~500 tokens a ~250 tokens, proporcionando ~1000 tokens de margen adicional contra el límite de la ventana de contexto

**Paso 2:** Reducido `SLIDING_WINDOW` de 10 a 5 en la segunda definición (línea 103)
- Impacto: mantiene consistencia entre ambas definiciones, eliminando el hazard de mantenimiento

**Paso 3:** Añadido retry consciente de 2013 en `call_minimax` (líneas ~461-469)
- Detecta `base_resp.status_code == 2013` ANTES de verificar `choices: None`
- Trunca el historial de conversación por `max(2, len(messages) // 4)` mensajes antes de reintentar
- Registra CRITICAL cuando 2013 dispara: `"[call_minimax] 2013 context window exceeded — truncating {N} messages ({len} -> {new}) and retrying"`
- Reutiliza el flag `choices_none_retried` existente para prevenir loops infinitos

**Paso 4:** Actualizado `_validate_sliding_window_and_timeout()` para enforcing SLIDING_WINDOW <= 5
- Cambio: límite de `> 10` a `> 5`
- Mensaje CRITICAL actualizado: `"SLIDING_WINDOW={window} exceeds maximum (5)"`

**Paso 5:** Creado archivo de pruebas `tests/test_minimax_context_window.py` con 5 tests
- Verifica que SLIDING_WINDOW = 5 en ambas líneas
- Verifica detección de 2013 en call_minimax
- Verifica truncamiento correcto del historial
- Verifica el log CRITICAL cuando 2013 dispara
- Verifica que el AST startup check pasar con ventana = 5

**Paso 6:** Reconstrucción del contenedor Docker
- Contenedor reconstruido y reiniciado
- Imagen: `minimax-agent-minimax-agent:latest`
- Estado: `Up ~36 segundos (healthy)`
- Health check verificado: `FailingStreak: 0`

## Resumen para Fernando

El problema era que el historial de conversación era demasiado grande para la ventana de contexto del modelo MiniMax-M2.7. Cuando el agente acumulaba 10 mensajes (~500 tokens) más el system prompt (~4000 tokens) más las definiciones de herramientas (~3000 tokens), el total de ~7500 tokens excedía el límite del modelo, causando errores 2013.

La solución fue reducir SLIDING_WINDOW de 10 a 5 mensajes Y añadir un mecanismo de retry que trunca automáticamente el historial cuando ocurre el error 2013, en lugar de reintentar con el mismo contexto guaranteed para fallar.

El contenedor está saludable y todos los tests pasan. Los errores de Odoo (HTTP 500/409) son problemas de datos/permisos del backend, no defectos del código del agente.
