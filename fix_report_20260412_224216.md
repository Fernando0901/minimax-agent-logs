## 3. Causa Raíz

**Archivo:** `/root/vision/proactive_monitor.py`
**Función:** `_validate_empty_status_handling()` (líneas 371-394)
**Ubicación de la llamada defectuosa:** línea 399

**Mecanismo técnico:**

```python
# Código defectuoso (antes):
try:
    asyncio.run(_validate_empty_status_handling())  # línea 399
except Exception:
    pass  # silencia todo — incluyendo el RuntimeWarning
```

La función `_validate_empty_status_handling()` era declarada como `async def` (coroutine asíncrono). Cuando `asyncio.run()` se llama al nivel de importación del módulo (no dentro de un contexto async existente), Python crea un nuevo event loop, envía la coroutine y **cierra el loop inmediatamente** antes de que la coroutine pueda ejecutarse. El Runtime de Python entonces emite:

```
RuntimeWarning: coroutine '_validate_empty_status_handling' was never awaited
```

Este warning se lanza como una excepción capturada por el `except Exception: pass`, lo que significa que:
1. El log `EMPTY_STATUS_VALIDATION_PASSED` **nunca aparece** en agent.log
2. La lógica de clasificación de tres niveles (`_classify_status()`) **nunca se verifica** al startup
3. Cualquier bug en la clasificación de estados vacío/no-vacío pasaría desapercibido

**Por qué no se detectó antes:** El `bare except Exception: pass` silenciaba el warning de forma efectiva — el contenedor seguía funcionando (solo con la validación saltada), así que el problema parecía intermitente.

## 4. Qué se Corrigió

### Cambio 1 — Convertir función a síncrona
```python
# Antes (línea 371):
async def _validate_empty_status_handling() -> None:

# Después:
def _validate_empty_status_handling() -> None:
```
La función solo llama a `_classify_status()` que es 100% síncrona — no hay I/O async, no hay await expressions. Convertirla a `def` elimina el problema del event loop por completo.

### Cambio 2 — Llamar directamente sin `asyncio.run()`
```python
# Antes (línea 399):
try:
    asyncio.run(_validate_empty_status_handling())
except Exception:
    pass

# Después:
try:
    _validate_startup_validation_sync()  # AST guard primero
    _validate_empty_status_handling()     # Llamada directa síncrona
except Exception as e:
    log.critical(f"[proactive] Startup validation failed: {e}")
    raise  # Re-raise — el contenedor falla rápido si la validación falla
```

### Cambio 3 —取代 bare except con manejo estructurado
El `except Exception: pass` que silenciaba todos los errores fue reemplazado por un re-raise después de un log CRITICAL. Ahora si la validación falla, el contenedor no arranca — fails fast.

### Cambio 4 — AST-based regression guard añadido
```python
def _validate_startup_validation_sync() -> None:
    """AST-based: verifica que _validate_empty_status_handling NO sea async def"""
    with open(__file__) as f:
        tree = ast.parse(f.read())
    for node in ast.walk(tree):
        if isinstance(node, ast.AsyncFunctionDef) and node.name == '_validate_empty_status_handling':
            raise AssertionError(
                "_validate_empty_status_handling must be sync (not async def) — "
                "async at module scope causes RuntimeWarning and silently skipped validation"
            )
```
Esta función corre al import del módulo y falla con `AssertionError` si alguien reintroduce `async def` en el futuro. Previene regresiones antes de que el RuntimeWarning pueda dispararse.

## Resumen Ejecutivo

El problema era que `_validate_empty_status_handling()` estaba declarada como función async pero se llamaba con `asyncio.run()` al nivel de importación del módulo — el event loop se cerraba antes de que la coroutine pudiera ejecutarse, causando un `RuntimeWarning` silenciado por el `except Exception: pass`. Esto hacía que la validación de startup nunca corriera, aunque el contenedor seguía funcionando normalmente.

El fix convierte la función a síncrona (`def`), elimina el `asyncio.run()`, reemplaza el bare except con manejo estructurado de errores, y añade un guard AST que falla inmediatamente si alguien reintroduce `async def` en el futuro. El contenedor ahora arranca saludable con 0 RuntimeWarnings y la validación de estados vacíos se ejecuta correctamente en cada startup.
