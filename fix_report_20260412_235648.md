## 3. Causa Raíz

### Explicación accesible para Fernando

Cuando el bot se inicia, hace una "sonda" de cada endpoint del microservicio de Odoo para verificar que estén funcionando. La sonda de `create_quotation` enviaba `partner_name: "test"` para identificar un partner de prueba.

El problema es que en Odoo, cualquier partner cuyo nombre **contenga** la palabra "test" (por ejemplo, "Test Company SA", "Testing Inc") coincidiría con esa búsqueda. Cuando hay **más de un match**, el microservicio correctamente rechaza la solicitud con HTTP 409 (Conflicto) porque no puede determinar cuál partner es el correcto.

Esto no es un bug en el código del bot ni del microservicio — es el comportamiento **correcto y deseado** para prevenir ambigüedades. El problema era que el valor de prueba `"test"` era demasiado genérico.

### Detalle técnico

**Archivo:** `/root/minimax-agent/agent/odoo_bridge.py`
**Función:** `_validate_odoo_bridge_endpoints()` → inline probe payload para `create_quotation`
**Línea:** 964

**Mecanismo:**
```python
# ANTES (problemático):
{"partner_name": "test", "lines": []}

# El microservicio en resolve_id_strict() hace:
ids = odoo.name_search("test")  # → múltiples IDs si existe algún partner con "test" en el nombre
if len(ids) > 1:
    raise HTTPException(status_code=409, detail="Ambiguous partner")  # ← HTTP 409
```

**Confianza:** 0.95

## 4. Qué se Corrigió

### Cambio principal
**Línea 964** de `odoo_bridge.py`:
```python
# ANTES:
{"partner_name": "test", "lines": []}

# DESPUÉS:
{"partner_name": "_probe_partner_xyz123", "lines": []}
```

El string `"_probe_partner_xyz123"` es extremadamente unlikely que coincida con cualquier partner real en Odoo — usa formato estilo UUID que no es típico en nombres de empresas.

## 5. Mejoras Estructurales Añadidas

### Capa de validación `_validate_probe_payload_uniqueness()`
Se añadió una nueva función de validación en tiempo de arranque que escanea **todos** los payloads de sonda en `VALID_PAYLOAD_ENDPOINTS` e `INLINE_PROBE_PAYLOADS` buscando strings genéricos en el campo `partner_name`:

**Patrones detectados:**
- `test`, `probe`, `sample`, `example`, `demo`, `temp`

**Comportamiento:**
- Si encuentra un patrón genérico → levanta `AssertionError` con mensaje descriptivo y bloquea el arranque
- Si todo está limpio → log: `PROBE_PAYLOAD_VALIDATION_PASSED — all partner_name values are unique`
- **No** bloquea strings estilo UUID que contengan "probe" como prefijo (ej: `_probe_partner_xyz123` es válido)

### Arquitectura de prevención de recurrencia
```
Payload genérico ("test") en sonda
        ↓
¿Se escapa de _validate_probe_payload_uniqueness()?
        ↓ (NO, falla en arranque)
AssertionError →_container_no_inicia
```

Esta validación es **AST-based**: analiza el código fuente en tiempo de importación, no hace llamadas de red. Es rápida, determinista y no depende del estado de Odoo.

**Solución:** Cambio a `partner_name: "_probe_partner_xyz123"` (string estilo UUID con probabilidad de colisión casi nula) + validación `_validate_probe_payload_uniqueness()` que detecta y bloquea patrones genéricos en todos los payloads de sonda.

**Resultado:** HTTP 409 eliminado. Container healthy (FailingStreak: 0). Suite de 6 tests pasando. La validación de payloads ahora previene que este error se repita en el futuro.

**Decisión del Pipeline:** KEEP — el fix se mantiene con confianza 0.97.
